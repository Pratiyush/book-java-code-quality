Fast feedback first Cheap checks (format, compile, unit) run pre-commit and on every PR. Developers never wait for the slow stage.Ch 33 key 75/79
New-code focus (clean-as-you-code) The PR gate blocks on quality of what the team touches — adoptable on legacy without a wall of existing findings.Ch 34 key 80
Un-bypassable enforcement Required status check + branch protection + merge queue means no path to main that skips the gate.Ch 35 key 81/82
Slow checks off the fast path SpotBugs, Sonar, SCA, and mutation run nightly — full rigor where it is affordable, not blocking every push.Ch 33 key 79
Why four stages, not one? A single gate that runs everything on every PR would be too slow to sustain. A gate that skips the slow checks on every PR would miss the deep findings. The feedback-latency ladder resolves this: fast checks give developers instant signal; slow checks give the team thorough signal on a schedule; enforcement is locked at the merge so neither is optional. The new-code focus (clean-as-you-code) is what makes this adoptable on a legacy codebase that already has thousands of findings — the gate only judges what the team changes.
