Scaling the manual craft from one method to ten thousand files across hundreds of repositories — without removing the safety net the craft requires

Concept — type-aware (LST), not text search

Lossless Semantic Tree (LST) OpenRewrite parses code into it

Every node carries full type information and the original formatting.

finds EVERY real reference to a type or method — even through implicit or aliased imports — with no false positives.

Preserves formatting, so the diff is the change and nothing else.

Regex / text search find-and-replace over characters

Matches text, with no model of types, scopes, or imports.

breaks subtly at scale — misses aliased references, hits unrelated matches; the errors the LST exists to prevent.

Using regex for a type-aware change is the wrong tool for the job.

Concept — automation proposes; tests and review dispose

Run the recipe over the LST, at estate scale
→
A pull request a proposal, not a fait accompli
→
Review the diff like any other change
→
CI gates verify tests + build
→
Merge once green & reviewed
Automation proposes — it scales the application of the craft. A recipe can be subtly wrong for the codebase it runs against.
Tests & review dispose — the output is something to verify, never to apply blind. Scaling the craft does not remove its verification.
Apply per recipe, verify incrementally. Large auto-generated diffs are hard to review (the size limits of Ch 37), so run one recipe, verify it, and merge — rather than one giant change. And reserve bulk automation for substantive, type-aware changes: mass-applying a stylistic recipe creates churn and noisy diffs (the standards-migration caution).
