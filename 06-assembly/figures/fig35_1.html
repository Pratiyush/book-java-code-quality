Why low agreement is good news. Lenarduzzi et al. (2021, arXiv:2101.08832) found “little to no agreement among the tools and a low degree of precision” studying six tools across 47 Java projects. Low cross-tool agreement means coverage is additive, not redundant — each tool flags mostly different things because each reads a different substrate.
Three failure modes of the naive stack. (1) Overlap: Checkstyle, PMD, and SonarQube each ship line-length, naming, and empty-block rules — running all three for the same concern multiplies noise. (2) Redundancy: two source-AST style checkers for the same concern cost build time with no coverage gain. (3) Noise: every tool has false positives; an un-tuned full stack as a build-breaker trains developers to ignore the gate.
SonarQube for IDE / Connected Mode (formerly SonarLint; renamed Oct 2024). Connected Mode pulls the team’s quality profile and gate down to the IDE so local findings match CI. Without a committed inspection profile + .editorconfig, “it’s clean in my IDE” is not a contract. Tune and baseline the full stack before gating it (Ch 19).
