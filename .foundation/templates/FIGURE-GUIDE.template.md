# FIGURE GUIDE — {{BOOK_SUBJECT}}

> **Figures are load-bearing**, carried per the per-chapter image budget in `GUIDELINES.md` §8 (`{{FIGURE_POLICY}}`). This guide governs how a figure is produced. The defining rule: **a designed diagram is authored as HTML, rendered by headless Chrome, and cropped to a PNG — never generated by an image model.** Authored HTML keeps every label exact and source-traced; that is the whole reason we render HTML instead of generating images. Referenced by `GUIDELINES.md` §8, `LEGAL-IP-RULES.md`, and Pipeline Steps 9a/9b; captured screenshots (technical profile — see BOOK-TYPE-PROFILES.md; book types without `{{EXAMPLE_POLICY}}` runnable artifacts drop the SCREENSHOTS section) come from the `DEMO-CATALOG.md` demo modules.

## What this guide covers — and what it does NOT

It governs two image kinds:

1. **Designed diagrams** — authored HTML rendered to a cropped PNG (this guide's main subject). Eight families, below.
2. **Captured screenshots** — native UI/tool surfaces shot from the running pinned artifact (the SCREENSHOTS section). *(technical/code profile — book types without runnable examples drop this kind.)*

It does **not** govern:

- **Code/quoted listings** — typeset bounded snippets (HARD rule 4). Book-typeset text, never screenshots of code. There are no "code screenshots"; a snippet a reader cannot copy is a defect.
- **Comparison tables / trade-off grids** — typeset Markdown tables, governed by the drafting standard.
- **Inline transcripts** — typeset as fenced code, not images.

## The HTML method (how every designed diagram is produced)

### Folder layout

```
05-figures/
  _assets/
    figures.css        the ONE house stylesheet — every diagram links it
    render.mjs         Puppeteer-core renderer (drives the installed Chrome)
    package.json       declares puppeteer-core; node_modules is gitignored
  NN_slug/
    figNN_x.html       the diagram, authored as HTML/CSS + inline SVG   ← source of truth
    figNN_x.png        rendered, cropped to #figure                      ← embedded artifact
    figNN_x.sources.md every label/number → pinned {{AUTHORITY_PIN}} source path  ← figure-accuracy gate
```

### Authoring a diagram

- One self-contained HTML file. The capture target is the element `id="figure"`. Link the shared stylesheet: `<link rel="stylesheet" href="../_assets/figures.css">`. Put only figure-local layout in a `<style>` block; palette and components come from `figures.css`.
- Build shapes from HTML/CSS boxes and inline SVG for arrows and curves. Use the shared component classes (`.step`, `.col-head`, `.phase-label`, `.note`, the `--ink/--line/--accent` variables) so all chapters share one look.
- **Text is authored, so it is exact.** Every label, arrow, fact, and number is real and traceable — bake it into the image. The caption (in the prose) carries the one-sentence idea; the labels carry the rest.

### Rendering

```
node 05-figures/_assets/render.mjs  <fig.html>  <fig.png>  "#figure"  3
```

- Drives the installed Google Chrome via puppeteer-core (no Chromium download). Screenshots **only the `#figure` element's bounding box** — "the required part" — at `deviceScaleFactor 3` for crisp print/ebook.
- One-time setup: `cd 05-figures/_assets && PUPPETEER_SKIP_DOWNLOAD=1 npm install`.
- **Commit both** the HTML (re-renderable source of truth) and the PNG (the asset the book embeds).

## House style (realized in `figures.css`)

- **Palette.** Clean and neutral — blues, grays, white, **one** restrained accent (`--accent`). No vendor colors, no decorative gradients.
- **Grayscale-safe.** Distinguish elements by lightness, border weight, and position — never hue alone. The diagram must read in black-and-white print.
- **Whitespace.** Generous. Clarity over polish.
- **Aspect.** Sized to the content (the renderer crops to the element). Landscape reads naturally for left-to-right flows; let the content set the shape.
- **No real faces. No vendor logos or trademarked marks** unless a direct comparison or migration topic requires one (`{{NEUTRALITY_STANCE}}`), and then only minimally (`LEGAL-IP-RULES.md`).

## Accessibility (alt-text required per figure — Step 9c A11Y)

Every figure carries a **text equivalent** so a screen-reader user, a low-vision reader, and a reader of the grayscale print get the same information a sighted reader of the color screen gets. The `accessibility-editor` authors this layer at Pipeline **Step 9c (A11Y)** and writes it into an `## Accessibility` block in the figure's `figNN_x.sources.md` sidecar. A11Y is **soft (FIX) at Step 9c but escalates to HARD at Step 15 (PRODUCTION-PROOF)** — a figure that reaches assembly without conformant alt-text fails the whole-book proof.

The per-figure checklist:

- **Alt-text (required, every figure).** One tight sentence stating what the figure *is and shows* — the same takeaway the caption gives a sighted reader, not "Figure 1.1" and not a pixel inventory. Every label it names matches the `figNN_x.html` source exactly and traces to the pin via the sidecar; it never names a label the figure does not carry.
- **Long-description (required for any spatial/flow/architecture diagram).** The full text equivalent — boxes and their relationships, arrow direction and order, columns/lanes, decision branches — so a reader who cannot see the diagram reconstructs the same mental model. Factual, in the `{{VOICE}}`; structure, not decoration.
- **Grayscale-safe (already the house rule, confirmed here).** Elements differ by lightness, border weight, label, or position — never hue alone (see House style). A distinction that survives only in color is an A11Y FIX.
- **Contrast + readable listings.** Text-to-background contrast is sufficient for low vision; *(technical profile — see BOOK-TYPE-PROFILES.md)* any code is real selectable text (a `// tag::` region a reader can copy), never a screenshot of code, and is understandable without syntax color.
- **Caption referenced before the figure** so a screen reader meets the reference first; alt-text, caption, and surrounding prose do not contradict each other.

Sidecar `## Accessibility` block shape (added beside the source-trace table):

```
## Accessibility
- Alt-text: <one-sentence equivalent>
- Long-description: <full structural description, in reading order>
```

## Source-trace (the figure-accuracy gate — HARD rule 3)

Each diagram ships a sidecar `figNN_x.sources.md` mapping **every depicted claim to a pinned {{AUTHORITY_PIN}} source path** — the same standard the prose meets. A figure depicts nothing the prose has not already traced. Comparison/benchmark numbers use pinned-source figures only; where the source gives no number, bars are sized **relatively** with no numeric axis, labeled "relative" — never an invented benchmark.

Sidecar shape:

```
# Figure source-trace — <title> (ch NN)
- Files: figNN_x.html → figNN_x.png. Family: <family>. Method: HTML → render → crop.
- One idea: <single sentence>.
- Verified against {{BOOK_SUBJECT}} {{AUTHORITY_PIN}}.

| Label in diagram | Claim | Pinned source ({{AUTHORITY_PIN}}) |
|---|---|---|
| <label> | <what it asserts> | <path within {{AUTHORITY_SOURCE}}> |

Source-trace status: YES / FLAGGED
```

## SCREENSHOTS (the captured family)

*(technical/code profile — see BOOK-TYPE-PROFILES.md; book types without `{{EXAMPLE_POLICY}}` runnable artifacts drop this whole section.)*

Native UI screenshots are the one image kind not authored as HTML. They are captured at Pipeline Step 4c from the **running pinned {{AUTHORITY_PIN}} companion module**.

- **Subject-native UIs only:** the tool surfaces shipped by {{BOOK_SUBJECT}} itself (e.g. its dev UI, API explorer, health UI, continuous-testing console).
- **Third-party-tool UIs are banned:** external dashboards, vendor consoles, registry/login pages. They carry vendor branding (neutrality), licensing exposure (IP), and they age with the vendor, not with the pin. **Replace each with a designed topology/flow diagram** (family 1, 2, or 6) — more neutral and more durable than a dashboard shot anyway.
- **Trace = a capture manifest** stored beside the PNG: `module + exact command + URL/route + what it shows + redactions`. Any reviewer can re-run the module and re-capture.
- **Reproducibility — one of two paths, never an undocumented ad-hoc shot:**
  - *Scripted* — a small headless helper (`05-figures/_assets/capture.mjs`: navigate to the route, wait for a named selector, screenshot at a fixed viewport), the screenshot analogue of `render.mjs`. Commit the helper invocation in the manifest.
  - *Manual* — a documented capture manifest under `05-figures/NN_slug/` recording `module + exact command + route + viewport + redactions`, complete enough that a reviewer reproduces the same shot by hand.
- **Hygiene:** redact secrets and hostnames, no real faces, crop to the load-bearing region.

## Referencing a figure in prose

- Reference a figure **before** it appears, with a verb that names what it does: "Figure 1.1 depicts the build-time-vs-runtime split…". (Reads as a teaching beat, not decoration.)
- **Numbering scheme: `Figure <chapter>.<n>`** — the chapter number, a dot, then the figure's ordinal within that chapter (the first figure in chapter 1 is `Figure 1.1`, the second `Figure 1.2`). Spell out "Figure" — no abbreviation ("Fig.").
- Caption in **sentence case, no terminal period**, carrying the one-sentence idea.
- **Caption template:**

  ```
  Figure <chapter>.<n> <one-sentence idea, sentence case, no terminal period>
  ```

- Never promise a figure the chapter does not contain.

## Neutrality rule (HARD rule 1)

Figures are neutral by the book's `{{NEUTRALITY_STANCE}}`. {{BOOK_SUBJECT}} is the subject. No competitor logos or branding unless a chapter is a direct comparison or migration, and even then minimal, neutral, never positioning any alternative as superior. Any cross-subject element in a diagram needs a cited {{AUTHORITY_SOURCE}} reference, exactly as the text does. Never add a figure to fill a quota; never crown a winner in an image.
