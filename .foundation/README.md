# `.foundation/` — the reusable book-production kernel

> A book-type-agnostic version of this project's gated, AI-driven authoring pipeline.
> Copy it, fill the placeholders for your subject, pick a **book-type profile**, and you have a
> running production pipeline for *any* book — technical, science, business, reference, or narrative.
> It is a **living kernel**: improve it over time and the improvements carry to the next book
> (see [`CHANGELOG.md`](CHANGELOG.md)).

This kernel is the distilled, generalized form of the Quarkus-book pipeline that lives in this repo.
The Quarkus book is **instance #1**; `.foundation/` is the **mould**.

## What the kernel gives you

A complete, gated production system, invariant across book types:

- **Five locked phases:** Foundation → Research → Select → Draft + Gate → Assemble.
- **One authority source, pinned.** Every fact traces to a frozen, citable source of truth (a code tag, an edition of the literature, a corpus). Nothing is invented.
- **A law hierarchy** — one top-level GUIDELINES file, with subordinate voice / neutrality / scoring / source-pin / legal-IP / pipeline files.
- **A scoring rubric** — five quality clusters (1–10) plus pass/fail content-floors and one ship bar.
- **Gate agents** — research, source-verify, draft, (build/verify), clarity, audit, score, reconcile, maintain — each owning one step.
- **Orientation discipline** — `AGENTS.md` (the map), a single source of live state (`LEDGER.md` §1), and a non-circular read order, so a fresh agent is never misled by a stale file.
- **Continuous improvement** — a learnings log and a promotion path into the law.

## What varies by book type (the only knobs)

Everything book-specific is a `{{PLACEHOLDER}}` token resolved from a **book-type profile** in
[`BOOK-TYPE-PROFILES.md`](BOOK-TYPE-PROFILES.md):

| Knob | Technical/code | Science | Business/how-to | Reference/cookbook | Narrative non-fiction |
|---|---|---|---|---|---|
| Authority source | pinned repo + tag | peer-reviewed literature | cited research + cases | official docs | primary sources / interviews |
| "Never invent" units | config keys, GAVs, signatures | claims, figures, citations | stats, case facts | API/option names | quotes, dates, events |
| Examples | runnable compiled modules | thought experiments | worked scenarios | recipes | — |
| Compile floor (FLOOR C) | **yes** (green build) | no | no | optional | no |
| Neutrality stance | framework-agnostic | balanced theories | vendor-neutral | tool-neutral | fair to subjects |
| Figures | HTML diagrams + screenshots | conceptual diagrams | frameworks/matrices | tables/listings | photos/maps (rights-cleared) |

## Layout

```
.foundation/
  README.md              ← this file
  INSTANTIATE.md         ← step-by-step: spin up a new book from the kernel
  BOOK-TYPE-PROFILES.md  ← the per-book-type parameter values
  CHANGELOG.md           ← versioned kernel improvements (keep improving over time)
  templates/             ← generalized law, pipeline, orientation, chapter/research/score templates (*.template.md)
  agents/                ← generalized gate-agent specs
  commands/              ← generalized command specs
  scripts/               ← portable pipeline scripts (source-pin, verify, lint, reconcile, …)
```

## Relationship to a live book project

A live book (like this Quarkus repo) is **not** generated by symlink — you **instantiate** the kernel:
copy the templates out, resolve the placeholders, delete the parts your book type doesn't use, and
from then on the book evolves on its own. When you learn something general (a better gate, a clearer
template, a sturdier script), fold it **back** into `.foundation/` and bump the kernel version so the
*next* book starts ahead. See [`INSTANTIATE.md`](INSTANTIATE.md).
