# A Gate the Team Keeps On

*Designing the CI pipeline that runs every gate in the book — ordered for fast feedback, scoped to new code, and kept fast enough that nobody routes around it · 75 (folds 76, 79) · Part IX (opener)*

> Wire up every gate you've built and run them all, blocking, on every pull request, and within a month the slow ones are skipped with `[skip ci]` and the strict ones are set to `continue-on-error`. The gates didn't fail. The pipeline did.

## Hook

A team has built every gate in this book. The analyzers from Part IV, the test suite and coverage and mutation gates from Part V, the architecture rules from Part VI, the dependency and supply-chain scans from Part VII, the security gate from Part VIII — each works, each was carefully tuned in isolation. Then they wire all of it into one CI pipeline, set everything to block, and run it on every pull request. The pipeline takes forty minutes and fails on three hundred pre-existing findings the PR never touched. Within a month, half the checks are set to `continue-on-error`, developers merge with `[skip ci]`, and the quality program is decorative. Not one gate was wrong. The *pipeline* was never designed — and an undesigned pipeline collapses under its own weight.

Designing the pipeline so the team *keeps it on* is the subject of Part IX, and it opens here. The previous parts built the individual gates; this part assembles them into one system that runs on every change without being slow enough or strict enough to get routed around — which, as Chapters 19 and 32 kept showing, is the way every gate dies. That assembly is three design decisions, one per section of this chapter. **Order** — which checks run when, so feedback is fast and failures are cheap (the pipeline). **Policy** — what *blocks* a merge versus what merely warns, scoped so a legacy codebase doesn't block every PR (the quality gate). **Performance** — how to keep the whole thing fast enough that nobody has a reason to skip it (gate speed). Each stage is a fitness function (Chapter 26); the pipeline is the portfolio; and getting these three right is the difference between quality that's enforced and quality that's theatre.

## Overview

**What this chapter covers**

- **Pipeline design**: fail-fast, cheap-to-expensive stage ordering, and the PR / main / nightly split for fast feedback.
- **Quality-gate policy**: what can block a merge, the new-code-versus-whole-repo decision (clean-as-you-code), and block-versus-warn.
- **Gate performance**: caching, incremental analysis, parallelism, and stage placement — speed as a quality concern.
- The unifying frame: the pipeline as a fitness-function portfolio, and the failure mode (a gate routed around) all three guard against.

**What this chapter does NOT cover.** The CI *platforms* (GitHub Actions, GitLab, Jenkins) and PR automation (reviewdog, Danger), and coverage *strategy* in depth — the next chapter. Branch protection, trunk-based development, merge queues, and pre-commit/local parity (the chapter after). Release quality (the chapter after that). DORA metrics in depth (Part X). The individual gates themselves (Parts IV–VIII — what *runs* in the pipeline). This is the **pipeline-design umbrella**; specifics route onward. CI platform and tool specifics are verified at the pin.

**If you hold one idea**, hold this: *a quality pipeline is a fitness-function portfolio, and it survives only if it's ordered for fast feedback, scoped to gate new code rather than the legacy mountain, and kept fast enough that developers never have a reason to route around it — order, policy, performance, all serving "a gate the team keeps on."*

## How it works

### Pipeline design: order for fast feedback

A CI pipeline is where every quality gate in this book actually runs, on every change, and its *design* — which checks, in what order, with what feedback latency — decides whether quality is enforced or theatre. The organizing principle is the one from the analyzer and build chapters, now applied to the whole program: **fail fast, cheapest and most-likely-to-fail first.**

| Stage | Checks | Why here |
|---|---|---|
| 1. Compile + format + fast lint | `javac`, format-check, Checkstyle/PMD (Ch 16) | seconds; catches the most common, cheapest failures first |
| 2. Unit tests + coverage | JUnit (Ch 21), JaCoCo (Ch 23) | fast feedback on behaviour |
| 3. Heavier static + security | SpotBugs/Error Prone/Sonar (Ch 16–17), SAST/SCA/secrets (Ch 28, 31) | slower analysis, still on the PR |
| 4. Integration tests | Testcontainers (Ch 22) | needs Docker; minutes |
| 5. Mutation / performance | PITest (Ch 23), JMH | expensive; often main/nightly |
| 6. Package + SBOM + sign | reproducible build, CycloneDX (Ch 28–29) | release artifacts |

> **CONCEPT** *Not everything runs on every pull request.* Fast feedback is the core principle — a pipeline that takes forty-five minutes to give a verdict is one developers learn to bypass. So the blocking PR path is kept short (the cheap stages), while the expensive checks (full DAST, deep mutation, long performance suites) run on **main** or **nightly**, where their latency doesn't block a developer waiting to merge. The PR answers "is this change cheaply, obviously broken?" in minutes; the slower stages answer the deeper questions out of the critical path.

Two properties make the pipeline trustworthy: it runs the *same* build developers run locally (`./mvnw -B verify`, Chapter 27, with pinned tool versions so it can't drift), and every gate's verdict is *recorded* on the pull request (PR decoration, next chapter) so a failure is immediately actionable rather than a mysterious red X. The frame to carry: this is the fitness-function portfolio of Chapter 26 made operational — each stage an automated, continuous assessment of one quality characteristic, the whole pipeline the deliberate set of them.

### Quality-gate policy: what blocks, and on whose code

A pipeline that *runs* checks still has to decide what to *do* when one fails — and that policy, not the checks themselves, is what most often makes or breaks the gate. A **quality gate** is the policy deciding whether a change can merge: which findings break the build versus merely warn. Too strict and it's bypassed; too loose and it's meaningless. A gate can check compilation, passing tests, a coverage threshold (Chapter 23), no new high-severity static or security findings, no new secrets, no banned dependencies or licenses (Part VII), architecture rules (Chapter 26), complexity thresholds — and SonarQube's Quality Gate packages many of these into one configurable policy.

> **CONCEPT** *Gate new code, not the whole repo — "clean as you code."* This is the breakthrough that made gates practical at scale. Gating *whole-repo absolutes* on a legacy codebase is impossible: a "zero findings" or "80% coverage" rule applied to a million existing lines blocks every pull request on debt the PR didn't create. Gating **new and changed code** instead — no *new* bugs, no *new* uncovered code, hotspots reviewed — makes the gate adoptable immediately and assigns ownership correctly: developers own the quality of what they *touch*. SonarQube's default gate is new-code-focused for exactly this reason; it's the same scoping that saved the static-analysis gate in Chapter 19 and the security gate in Chapter 32, now stated as the general policy.

The second policy axis is **block versus warn**: block on objective, low-false-positive, high-severity *new* findings; warn and triage on the subjective or noisy ones (Chapter 19), so the gate's red light always means something real. Security findings often route to a reviewer rather than auto-blocking (Chapter 32). The gate is *enforced* by becoming a **required status check** in branch protection (next-but-one chapter), so it genuinely can't be merged around — and the healthy complement to that hard enforcement is a *documented, tracked override path* (with justification and approval) for the rare legitimate exception, because a sanctioned escape hatch is far healthier than developers quietly disabling the check when it blocks something it shouldn't.

The honest limits are the failure modes the policy must avoid. *Too strict gets bypassed* — the number-one failure, and a gate the team routes around is a net negative. *Gates get gamed* (Goodhart's law, the metrics folklore): gate on coverage percentage and you get assertion-free tests (Chapter 23), because a gate measures a proxy, not the quality itself. And a *green gate is not good code* — it means "no detected policy violations," and design and logic quality still need human review (Chapter 84).

### Gate performance: fast enough to keep on

The third decision is the one teams treat as an afterthought and shouldn't: **gate speed is a quality concern.** A pipeline too slow for fast feedback gets bypassed; a slow local build makes developers skip the checks before pushing. Speed is the precondition for every other gate in the book — make the gate fast and enforcement and developer velocity reinforce each other (the DORA finding); make it slow and they fight, and the gate loses. The levers:

- **Caching** — Gradle's build and configuration caches (local and remote), Maven's reactor and incremental build, and CI dependency caching (`~/.m2`) so unchanged work isn't re-downloaded or rebuilt.
- **Incremental analysis** — run analyzers on changed modules/files, and Sonar's PR analysis on new code, rather than re-scanning the whole repo every PR.
- **Parallelism** — Maven `-T` for parallel modules, Gradle's parallel workers, JUnit parallel execution, and CI job sharding (splitting tests across runners).
- **Stage placement** — the PR/main/nightly split from the pipeline section *is* a performance lever: keep the blocking PR path short, defer the expensive checks.
- **Test-suite speed** — fast unit tests in the inner loop, integration tests later, and quarantine or fix the flaky tests (Chapter 20) that waste retries and erode trust.

> **CONCEPT** *Speed has its own honest limits.* Each lever can backfire. **Caching can mask staleness** — a bad cache key skips work that should run and produces a false green, and cache invalidation is famously hard, so verify cache correctness and run clean builds on main. **Parallelism surfaces flakiness** — tests with hidden shared state that passed serially fail under parallel execution (Chapter 20), which is a real bug the speedup exposed, not a regression. **Incremental analysis can miss cross-module effects** — a change whose impact lands outside the changed file — so periodic full scans (nightly) backstop it. And optimization has diminishing returns: an over-tuned build is itself hard to maintain, so measure where the time actually goes (pipeline duration is a meta-quality metric; a creeping build time is debt) and optimize the real bottleneck, not a guess.

## Deep dive: three decisions, one goal — a gate nobody routes around

The three sections are not independent topics; they are three facets of a single design goal, and naming it is what turns "set up CI" into engineering. The goal is **a gate the team keeps on** — and every gate in this book, across every part, has died the same two deaths when that goal wasn't met: too *slow*, so developers bypass it, or too *strict*, so they disable it. Ordering, policy, and performance are precisely the three levers against those two deaths. Performance attacks "too slow" directly. Policy (clean-as-you-code, block-narrowly) attacks "too strict" directly. And ordering serves both — fast feedback from cheap-first staging, and a short blocking path that keeps the strict checks off the critical path. The three interlock: a perfectly-ordered pipeline with a too-strict policy still gets disabled; a lenient policy on a forty-minute pipeline still gets bypassed; you need all three.

This also resolves a tension that has run through the whole book: enforcement versus velocity. Naively, every gate you add slows the team and constrains them, so quality and speed look opposed — and a badly-designed pipeline makes that real, which is exactly why teams under deadline pressure strip gates out. But the DORA research found the opposite at well-run organizations: speed and stability *correlate*, because a fast, credible gate lets a team merge small changes frequently with confidence, while the absence of gates forces large, risky, infrequent releases. The reconciliation is in this chapter's three decisions: clean-as-you-code policy means the gate constrains only the new code a developer is already thinking about (low friction); fast performance means the verdict comes back before they've context-switched (low latency); and fail-fast ordering means they hear about the cheap mistake in seconds, not the expensive one in an hour. A well-designed pipeline doesn't trade quality against speed — it's the mechanism by which a team gets both, which is why "the pipeline was never designed" is the root cause when a team feels quality gates are slowing them down.

The honest center, shared with every gate chapter: **the pipeline enforces policy; it does not create quality, and it cannot fix culture.** A green pipeline means "no detected policy violations on new code" — it is necessary, high-value, and the thing that makes quality scale beyond individual heroics, but it is not a proof of good code (design and logic flaws pass every automated gate, Chapter 84) and it is helpless against a team that rubber-stamps red builds or normalizes `[skip ci]` (Chapter 1's culture point). More stages do not help such a team; fewer, well-chosen, fast, credibly-blocking stages do. The pipeline is the portfolio of fitness functions made to run continuously and cheaply enough that the team treats its verdict as real — and that, far more than the number of gates, is what determines whether the quality program built across this entire book actually holds.

## Limitations & when NOT to reach for it

- **A slow pipeline gets bypassed.** If PR feedback is too slow, teams merge around it; speed is a first-class design concern, not an afterthought. Keep the blocking PR path short and defer expensive checks.
- **A too-strict gate gets disabled.** The number-one failure; a gate the team routes around is a net negative. Scope to new code (clean-as-you-code) and block narrowly, or watch it get turned off.
- **Gates get gamed.** Gate on a proxy (coverage %) and the team optimizes the proxy (assertion-free tests), not the quality (Goodhart's law). Gates measure proxies; pair them with review.
- **A green pipeline is not good code.** It means "no detected policy violations" — design, logic, and authorization quality still need human review (Chapter 84). Don't read green as a quality proof.
- **More stages ≠ better.** Every gate adds latency, maintenance, and false-positive surface (Chapter 19); include only gates that earn their cost.
- **Caching can produce false greens.** A bad cache key skips work that should run; verify cache correctness and run clean builds on main. Cache invalidation is genuinely hard.
- **Parallelism exposes flaky tests.** Hidden shared state fails under parallel execution — a real bug, not a regression, but it must be fixed (Chapter 20) before parallelism pays off.
- **CI can't fix a broken culture.** A team that rubber-stamps red builds or `[skip ci]`s isn't helped by more gates (Chapter 1); the pipeline enforces policy, it doesn't create the will to care.
- **The pipeline config is code.** It rots, drifts, and needs ownership and review like any code; an over-tuned build is its own maintenance liability.

## Alternatives & adjacent approaches

- **SonarQube Quality Gate** — packages many checks (new-code coverage, no new issues, hotspots reviewed) into one configurable, maintained policy; the common operationalization of this chapter's gate.
- **PR / main / nightly split** — the structural alternative to "everything on every PR"; the primary lever for fast feedback without dropping coverage of the expensive checks.
- **Merge queues and trunk-based development** (next-but-one chapter) — re-validate against the latest base and keep the gate meaningful as the codebase moves under it.
- **Pre-commit hooks** (next-but-one chapter) — push the cheapest checks even further left, before CI, with the local↔CI parity caveat.
- **Documented override paths** — the disciplined alternative to quiet gate-disabling: a tracked, justified, approved exception is far healthier than a silent `continue-on-error`.

These compose into one enforceable system: a fast, fail-fast pipeline; a clean-as-you-code blocking policy; performance levers to keep it quick; and the branch-protection enforcement (next chapters) that makes the gate unbypassable while leaving a sanctioned escape hatch.

## When to use what

- **To order the pipeline:** fail-fast, cheap-to-expensive — compile/format/lint, then unit+coverage, then heavy static/security, then integration, then mutation/perf, then package.
- **To keep PR feedback fast:** block on the cheap stages at the PR; defer full DAST, deep mutation, and long performance suites to main or nightly.
- **To make the gate adoptable on a legacy codebase:** clean-as-you-code — gate new/changed code, not whole-repo absolutes.
- **To set block-vs-warn:** block on objective high-severity new findings; warn and triage the subjective and noisy; route security to a reviewer.
- **To enforce it:** a required status check in branch protection, with a documented, tracked override path.
- **To keep it fast:** caching, incremental analysis, parallelism, and the PR/main/nightly split — verifying cache correctness and fixing the flakiness parallelism exposes.
- **To know if it's healthy:** measure pipeline duration as a metric; a creeping build time is debt to pay down.
