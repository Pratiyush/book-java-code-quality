## Accessibility

- Alt-text: The five test doubles ordered Dummy, Fake, Stub, Spy, Mock from inert to interaction-checking, split into state versus behaviour verification.

- Long-description: The figure, titled "Fig 21.2 — The five test-double taxonomy" and subtitled with the Fowler/Meszaros spectrum from Dummy to Mock and its hinge of state verification (stub) versus behaviour verification (mock), opens with a five-card spectrum bar shaded progressively darker from left to right. The cards, in order, with each definition quoted verbatim from Fowler's "Mocks Aren't Stubs": Dummy — passed around, never actually used ("just used to fill parameter lists"; example null / new DummyLogger()); Fake — a working implementation with a shortcut, not production-suitable ("take some shortcut which makes them not suitable for production"; example a HashMap-backed repository); Stub — returns canned answers to calls made during the test ("provide canned answers to calls made during the test"; example when(catalog.price(id)).thenReturn(Money.of(90))); Spy — a stub that also records how it was called, a partial real ("stubs that also record some information based on how they were called"; example @Spy / Mockito.spy(real)); Mock — pre-programmed with call expectations, a behavioural specification ("pre-programmed with expectations which form a specification of the calls they are expected to receive"; example verify(gateway).charge(eq(id), any())). A direction bar under the cards runs from "inert placeholder" on the left to "interaction-checking" on the right.

  Below the spectrum sit two side-by-side cards naming the hinge. State verification: the test checks the outcome — the state of the system after the method runs; used with stubs and fakes, the doubles supply inputs and the assertion checks outputs; applies to Dummy, Fake, Stub (and Spy used as a stub); its strength is that tests survive refactoring of internal call structure because the assertion is on observable behaviour. Behaviour verification: the test asserts an interaction — that the unit made the right calls on its collaborator — with Fowler's verbatim line "Mock objects always use behaviour verification, while a stub can go either way"; applies to Mock (and Spy when verifying calls); its cost is that it couples the test to the implementation's call structure and breaks on a refactor that changes how the code collaborates even when observable behaviour is unchanged.

  At the bottom, a strip titled "Mockito 5.23.0 — mechanism (STRICT_STUBS default via MockitoExtension)" has four cells: Create — @Mock, @Spy, @Captor, @InjectMocks via @ExtendWith(MockitoExtension.class); Stub (for stub/spy) — when().thenReturn(), when().thenThrow(), doReturn().when() for void or spy; Verify (for mock) — verify(m, times(1)), verify(m, never()), verifyNoMoreInteractions(); Quality guard — STRICT_STUBS (default), an unused stub raising UnnecessaryStubbing and an argument mismatch raising PotentialStubbingProblem. A closing note states that MockitoExtension defaults to Strictness.STRICT_STUBS so a dead stub fails the build (UnnecessaryStubbingException) and an argument mismatch on a stubbed call raises PotentialStubbingProblem, and that value objects (records / immutable data) should not be mocked — use the real instance.
