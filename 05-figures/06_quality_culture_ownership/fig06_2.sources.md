## Accessibility

- Alt-text: Six-stage lifecycle timeline from IDE to production where the cost to fix a defect rises left to right; shifting left keeps the fix cheap.
- Long-description: A left-to-right timeline of six lifecycle stages, each a box with the tools that run there and a fix-cost label, with the header fills ramping from light on the left to dark on the right to signal rising cost. Stage 1, IDE / local: inspection at type-time, compiler errors, local formatter (IDE lint, Checkstyle) — fix cost lowest. Stage 2, pre-commit: a hook runs before code leaves the machine (pre-commit hook, Spotless) — fix cost low. Stage 3, compile-time: annotation processors, null-safety, type errors (Error Prone, NullAway) — fix cost low to medium. Stage 4, fast CI: unit tests, static analysis, coverage gate (PMD, SpotBugs, JaCoCo) — fix cost medium. Stage 5, integration / review: integration tests, PR review, quality gate (SonarQube gate, ArchUnit) — fix cost high. Stage 6, production: defect reaches users, incident, rollback or hotfix (incident / hotfix) — fix cost highest. Below the stages, a cost bar runs from "Low cost" on the left to "High cost" on the right. A direction row carries a "Shift left" label with a left-pointing arrow on the left and "pre-shift-left: defects found here" on the right. A lineage note credits W. Edwards Deming (build quality into the process rather than inspecting it at the end) and Larry Smith's "shift-left testing" (Dr. Dobb's, 2001), and states that culture decides whether developers welcome that feedback or route around it.
- Grayscale-safe: Yes. Rising cost is encoded by the light-to-dark lightness ramp on the stage headers and by the explicit "Fix cost: lowest…highest" text labels and the lightness gradient of the cost bar — never by hue. The two darkest stage headers use white text on dark fill, which still reads. The "Shift left" pill is in the accent color but carries its own text label.
- Contrast: Sufficient. Dark ink on light fills for the lighter stages; white text on the dark fills of the two rightmost stages reads clearly at print size.
- Code legibility: The inline tool tokens (for example pre-commit hook, NullAway) are real authored monospace text, not a screenshot, and read without syntax color.
