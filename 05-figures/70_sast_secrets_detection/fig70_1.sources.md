## Accessibility
- Alt-text: Three side-by-side security lenses — SAST (your code, static), SCA (your dependencies), DAST (the running app) — each with what it catches and its blind spots, none a substitute for the others.
- Long-description: The figure sets three lens columns side by side, separated by dividers labelled "layered, not substitutes". The first lens, SAST (Static Application Security Testing, marked as this chapter's subject), analyzes the project's own source or bytecode without running it and finds injection, XSS, path traversal, and crypto misuse — the classes the previous chapter taught developers to design out. Inside SAST, a "two detection techniques" block contrasts pattern matching (syntactic rules that flag known-bad API calls, for example Cipher.getInstance("AES") or MD5) with dataflow/taint analysis (traces an untrusted source such as a request parameter through the program to a dangerous sink such as a SQL string or Runtime.exec, stronger for injection and catching flow across multiple method calls). SAST catches well: SQL injection, XSS, path traversal, crypto misuse; attacker-controlled source-to-sink flows; and known-bad API calls. Its blind spots: authorization and business-logic flaws (no source-to-sink flow), false positives on unreachable sinks, and slow deep dataflow on large codebases. Its tools, crowning none, are FindSecBugs, Semgrep, CodeQL, Snyk Code, and SonarQube (commercial taint). The second lens, SCA (Software Composition Analysis, owned by Part VII / Chapter 28), analyzes the project's dependencies, matching dependency GAV coordinates against vulnerability databases (CVE/OSV/GHSA), and finds vulnerable libraries SAST will not flag; it catches known CVEs in third-party and transitive dependencies and license risks, and is blind to code the team wrote, reachability, and runtime configuration; its tools and SBOM are deferred to Part VII (Chapter 28, key 65). The third lens, DAST (Dynamic Application Security Testing, owned by Chapter 32), probes the running application from outside like an attacker, sending malformed/attack inputs to a live app and finding runtime misconfigurations neither static tool can see; it catches runtime misconfigurations, auth/session flaws visible at runtime, and infrastructure exposures, and is blind to code-level source/sink paths and to anything requiring shift-left; its tools are deferred to Chapter 32 (key 73). A bottom note states the triad is layered, not either/or — SAST sees a dangerous sink SCA cannot, SCA sees a vulnerable library SAST will not flag, DAST sees a runtime misconfiguration neither static tool can find, and a security program runs all three. A routing strip points to the chapters that own each topic.
- Grayscale/contrast/legibility note: Grayscale-safe — the three lenses are distinguished by column position and bordered headers, not by hue, and the SAST focus is marked by an accent top-border whose meaning ("this chapter") is also stated in its label. Within each lens the "catches well" and "blind spots" sections use green and red header text respectively, but each is also spelled out in plain uppercase words ("CATCHES WELL" / "BLIND SPOTS") and separated by position, so the distinction reads in black-and-white print. All tool pills and code tokens (Cipher.getInstance, Runtime.exec) are real, selectable monospace text understandable without syntax color; contrast against the light tint fills is sufficient for low vision.
