Fig 70.1 — SAST, SCA, and DAST: three security lenses, layered

Each analyzes a distinct surface; none substitutes for the other two. SAST uses two complementary techniques.

SAST  —  this chapter
Static Application
Security Testing
Analyzes your project’s own source/bytecode — without running it
Finds injection, XSS, path traversal, crypto misuse. Catches the vulnerability classes Chapter 30 taught developers to design out.
Two detection techniques
Pattern
matching
Syntactic rules; flags known-bad API calls and dangerous forms.
e.g. flag Cipher.getInstance(“AES”), flag MD5
Dataflow /
Taint analysis
Traces untrusted source (e.g. request parameter) through the program to a dangerous sink (e.g. SQL string, Runtime.exec). Stronger for injection; catches the flow across multiple method calls.
e.g. user input → 3 methods → SQL query string without parameterization
catches well
  • SQL injection, XSS, path traversal, crypto misuse
  • Attacker-controlled source → dangerous sink flows
  • Known-bad API calls (pattern-based)
blind spots
  • Authorization & business-logic flaws (no source→sink flow)
  • Unreachable sinks may produce false positives
  • Deep dataflow is slow on large codebases
FindSecBugs Semgrep CodeQL Snyk Code SonarQube (commercial taint)

not

SCA  —  Part VII (Ch 28)
Software Composition
Analysis
Analyzes the project’s dependencies — libraries and transitive deps
Matches dependency GAV coordinates against vulnerability databases (CVE/OSV/GHSA). Finds vulnerable libraries SAST will not flag.
catches well
  • Known CVEs in third-party libraries
  • Transitive dep vulnerabilities
  • License risks in dependencies
blind spots
  • Code the team wrote (SAST’s territory)
  • Reachability of vulnerable code path
  • Runtime configuration flaws
tools & SBOM → Part VII (Ch 28, key 65)
 
DAST  —  Ch 32 (key 73)
Dynamic Application
Security Testing
Probes the running application from outside — like an attacker
Sends malformed/attack inputs to a live app and observes responses. Finds runtime misconfigurations neither static tool can see.
catches well
  • Runtime misconfigurations
  • Auth/session flaws visible at runtime
  • Infrastructure-level exposures
blind spots
  • Requires a running app (not shift-left)
  • Code-level source/sink paths (SAST’s territory)
  • Dep vulnerabilities in unprobable paths
tools & DAST → Ch 32 (key 73)
Layered, not either/or. SAST sees a dangerous sink SCA cannot; SCA sees a vulnerable library SAST will not flag; DAST sees a runtime misconfiguration neither static tool can find. A security program runs all three; no lens substitutes for the other two.
