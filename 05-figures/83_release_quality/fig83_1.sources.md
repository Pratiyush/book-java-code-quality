## Accessibility
- Alt-text: A four-stage release pipeline drawn as a closed loop where shift-left gates and shift-right release quality feed each other.
- Long-description: A horizontal band across the top marks two directions, both reading left to right: the left half "SHIFT-LEFT — catch defects before merge (the gates of Ch 33–35)" and the right half "SHIFT-RIGHT — limit damage and learn in production (release quality, this chapter)". Below it, four stage cards run left to right, each a step darker in fill, joined by forward arrows. Stage 1 "Release gates": all CI gates green on the release commit (Ch 33–35); SBOM generated and artifact signed/attested (SLSA/cosign, Part VII); semver bump plus changelog; smoke tests against a staged build — final artifact-level checks, not re-litigating code quality. Arrow "green & signed" to Stage 2 "Progressive delivery": canary (small % of traffic, watch error/latency, promote or roll back); blue-green (two envs, rollback is a traffic switch); feature flags (deploy dark, release gradually, kill-switch, Ch 35); a highlighted badge reads "a defect slips → hits 1%, not 100%". Arrow "gradual rollout" to Stage 3 "Production & monitor": error tracking (Sentry-class) surfaces a regression; metrics/alerts plus SLOs (Part XIII); continuous CVE re-scan of shipped deps (Dependency-Track, Part VII). Arrow "regression found" to Stage 4 "Promote or roll back": healthy promotes to 100%; bad triggers a flag flip / traffic switch / rollback; a stateful change needs a backward-compatible (expand–contract) migration, not raw rollback. Below the pipeline, a concept strip states "Decouple deploy from release": deploying code and releasing a feature are separate acts, so code deploys continuously dark behind flags while features release gradually — frequent and safe are produced by the same mechanism, not traded off (DORA, key 02). Beneath it, a feedback panel headed with a loop arrow shows the chain "a production regression becomes FIX the bug + ADD a test (Ch 20) + sometimes a NEW GATE (a fitness function, Ch 26)", and a footer notes the new gate is itself shift-left for the next change, returning the arc to Stage 1 — the quality cycle closes. A measurement strip notes DORA stability metrics (change-failure rate and recovery time) measure release quality and that canary, flags, and rollback drive them down (Part X, key 85). A final dashed "honest ceiling" strip lists six limits in two columns: safe release is not good code; it needs infrastructure and good metrics; flags become debt without a removal discipline; rollback is not always clean for stateful changes; feedback only helps if acted on; and gates verify the artifact, not the design (the logic flaw is caught upstream by review, Ch 37).
