## Accessibility
- Alt-text: A seven-step left-to-right remediation playbook, with a churn-by-pain note and two rejected anti-patterns.
- Long-description: The figure reads left to right as seven numbered stages joined by arrows. Stage 1 "Assess & baseline" (marked as the start with the warm accent) stands up the toolchain in report-only mode, baselines existing findings, and measures debt, coverage, complexity, and hotspots (Ch 3 / Ch 38 / Ch 2). Stage 2 "Stop the bleeding" turns on new-code gates (clean-as-you-code), so no new debt accrues and the curve bends (Ch 33–34). Stage 3 "Establish a safety net" adds characterization tests, approval testing, and seams before any change (Ch 39 / Ch 24). Stage 4 "Prioritize by churn × pain" remediates hotspots — high-change × high-complexity — first, declining to fix everything (hotspots / debt; Ch 1). Stage 5 "Pay down incrementally" refactors in small green steps, automates the bulk (pointing to Figure 40.2), modernizes the Java version where it unblocks, and ratchets thresholds (Ch 39 / OpenRewrite / Ch 38). Stage 6 "Replace the unsalvageable" uses the strangler fig (Ch 39, Fowler 2004). Stage 7 "Sustain" shows the trend on dashboards and builds culture and ownership (Ch 38 / Ch 1). A phase band groups steps 1–3 (stop it getting worse, make change safe), steps 4–6 (pay down where it hurts), and step 7 (sustain over months). Below, a concept note states that debt in frozen code accrues no interest while the same complexity in a weekly-edited file costs continuously, so remediation targets hotspots rather than the worst absolute metric. Two red boxes each headed with an "X" state the playbook's rejections: never a big-bang rewrite (it overruns, reintroduces debt, and chases a moving target — attributed to Fowler, not a universal law) and never a baseline without a paydown plan (a register nobody works is formalized ignoring; pair every baseline with the hotspot paydown and a ratchet).
