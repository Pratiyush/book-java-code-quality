## Accessibility
- Alt-text: A three-band diagram of the automation engine — a top contrast of OpenRewrite's type-aware Lossless Semantic Tree (finds every real reference, preserves formatting) against regex/text search (breaks subtly at scale); a middle map of three tools in three niches, crowning none (OpenRewrite for large migrations, Refaster for compiler-integrated pattern fixes, IDE structural refactors for interactive single-codebase changes); and a bottom left-to-right flow showing a recipe producing a pull request that review and CI gates verify before merge — automation proposes, tests and review dispose.
- Long-description: The figure has three stacked bands. The top band, "type-aware (LST), not text search," sets two cards side by side. The left card (with the warm accent) is the Lossless Semantic Tree: OpenRewrite parses code into it, every node carries full type information and the original formatting, so a transformation finds every real reference — even through implicit or aliased imports — with no false positives, and preserves formatting so the diff is only the change. The right card is regex / text search: it matches text with no model of types, scopes, or imports, so it breaks subtly at scale, missing aliased references and hitting unrelated matches — the errors the LST exists to prevent — and using it for a type-aware change is the wrong tool. The middle band, "three tools, three niches — the book crowns none," lists OpenRewrite (large cross-cutting migrations: composable declarative + imperative recipes over the LST via a Maven/Gradle plugin, large catalog, composite recipes; pinned 8.81.0 with rewrite-maven-plugin 6.38.0 and rewrite-gradle-plugin 7.32.0), Refaster (compiler-integrated pattern fixes: example-based, with a compilable @BeforeTemplate and @AfterTemplate; from Google, part of Error Prone, Ch 18), and IDE structural refactors (interactive single-codebase changes: IntelliJ structural search/replace plus the automated refactorings of Ch 39). A note states these are niches, not a ranking, composed by the scale of the change, and using the wrong one causes errors. The bottom band, "automation proposes; tests and review dispose," is a left-to-right flow: run the recipe over the LST at estate scale → a pull request (a proposal, not a fait accompli) → review the diff like any other change → CI gates verify (tests + build) → merge once green and reviewed. Two role boxes underline the split: automation proposes (it scales the application of the craft, and a recipe can be subtly wrong), while tests and review dispose (the output is to verify, never to apply blind). A closing note advises applying per recipe and verifying incrementally because large auto-generated diffs are hard to review (Ch 37 size limits), and reserving bulk automation for substantive, type-aware changes rather than stylistic recipes that create churn.
