What the edge guarantees
Single-thread ordering only
Does not cross thread boundaries. No cross-thread visibility guarantee on its own.
Mutual exclusion + full visibility
All writes by the releasing thread are visible to the acquiring thread. The heavyweight, complete tool.
Visibility — not atomicity
Latest write is visible to subsequent reads. Does NOT make read-modify-write atomic. count++ on a volatile field still loses updates (SpotBugs VO_VOLATILE_INCREMENT).
Publication at thread birth
Everything the parent thread did before start() is visible inside the new thread from its first action.
Publication at thread death
Everything the joined thread did is visible to the joining thread after join() returns.
Library-encoded edges
Every j.u.c utility documents its happens-before guarantee. Use the building block; do not re-derive by hand.

Transitivity If hb(A, B) and hb(B, C), then hb(A, C) — edges compose. A chain of synchronized blocks, a volatile read after a lock, or any sequence of j.u.c operations can establish a transitive happens-before path across many actions (JLS §17.4.5).
final-field freeze (§17.5)
A thread that sees a reference to a fully-constructed object is guaranteed to see correctly initialized values for that object's final fields. A record (all-final components) is thread-safe to share for free.
Trap: if this escapes during construction, the freeze is voided.
Unsafe publication (the silent hazard)
Handing a reference through a plain (non-final, non-volatile, un-locked) field lets a reader observe default values (0 / null), even after the constructor returned.
JMM permits this. Use one of the four idioms above.
volatile ≠ atomic
Use volatile for a flag or a published reference, never for a counter. Use AtomicInteger / LongAdder for read-modify-write.
SpotBugs VO_VOLATILE_INCREMENT catches volatile++ at build time.
