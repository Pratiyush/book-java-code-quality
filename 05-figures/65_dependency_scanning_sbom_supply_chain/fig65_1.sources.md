## Accessibility
- Alt-text: A five-stage supply-chain pipeline — Build, Generate SBOM, SCA Scan, Sign + Attest, Publish — bracketed by the three questions it answers, plus vulnerability-database and SLSA-level sidebars and honest-limit cells.
- Long-description: Across the top, a foundation bar states the precondition from the previous chapter: a deterministic build (pinned, converged, reproducible dependency graph), because a scan or SBOM is only as trustworthy as the build that produced it. Below it, five stages connect left to right with arrows. Stage 1, Build (the start stage), uses Maven or Gradle and resolves direct and transitive dependencies. Stage 2, Generate SBOM, produces CycloneDX 1.6 or SPDX (ISO/IEC 5962:2021) via the cyclonedx-maven-plugin or Syft, capturing all components including transitive ones. Stage 3, SCA Scan (CI gate), matches components against NVD, OSV, and GitHub Advisory using OWASP Dependency-Check 12.2.2, Grype ~0.108, Trivy 0.71.0, or Snyk, failing the build on a CVSS threshold. Stage 4, Sign + Attest, records an in-toto attestation and signs with Sigstore cosign using CI OIDC keyless identity, verified at deploy. Stage 5, Publish, ships the artifact and SBOM with the release (a later release gate, key 83). Beneath the stages, three brackets map each stage span to the question it answers: Q1 "Is anything known-vulnerable?" (SCA answers this) spans Build, Generate SBOM, and SCA Scan; Q2 "What ships?" (SBOM answers this) spans Generate SBOM; Q3 "Can I trust how it was built?" (Provenance / SLSA v1.0) spans Sign + Attest and Publish. A dashed continuous-monitoring loop notes that with Dependency-Track, new CVEs that land after ship trigger continuous re-scanning of published SBOMs against NVD and OSV, with update bots raising a fix PR, because a one-time CI gate is not enough. A right sidebar lists the vulnerability databases (NVD, CVEs CPE-matched by Dependency-Check; OSV, open-source-focused and ecosystem-aware; GitHub Advisory, curated package-ecosystem coverage) and notes that coverage and lag differ, some teams run two scanners, and a CVE not yet in a database is invisible to SCA. A second sidebar gives the SLSA v1.0 build-integrity levels — L1 provenance exists (signed), L2 hosted build service that is non-falsifiable, L3 hardened hermetic build platform — adding that it is a roadmap, not a single tool, and most teams start at L1. A bottom strip of four honest-limit cells warns: SCA covers known vulnerabilities only (zero-days and first-party bugs are out of scope, addressed by SAST and secure coding in Part VIII); vulnerable does not equal exploitable (CPE matching is fuzzy and flags CVEs in unused paths, narrowed by reachability analysis, requiring suppression with justification); an SBOM is inventory, not defense (it enables a fast response but fixes nothing, and misses shaded, dynamically-loaded, or vendored components); and a compliance driver (US EO 14028 and the EU Cyber Resilience Act make SBOMs mandatory in many sectors, turning supply-chain hygiene into a legal obligation).

The honest-limit cells use a warm warning tone on their titles, but each title is also prefixed with the warning glyph and a plain-text label, so the caution survives in print. The start-stage accent border is decorative; its "Build / start here" meaning is carried by position and the title. All tool names, versions, format identifiers, and standards references are real, selectable text at adequate print size; there is no code listing to police.
