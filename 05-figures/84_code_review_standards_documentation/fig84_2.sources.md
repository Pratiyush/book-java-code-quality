## Accessibility
- Alt-text: A three-by-two matrix showing that code review, coding standards, and documentation are each one move — the bot does the mechanical work (left column) so the human does the substantive work (right column) — with a shared note that automation enforces presence, never quality.
- Long-description: A left rail lists three practices, one per row: code review, coding standards, and documentation. Two columns follow, joined by a left-to-right arrow. The left column, "Mechanical → automated (the bot does it — cheap, deterministic, at scale)," holds: for review, bots check style, lint, and coverage before a human looks; for standards, a deterministic formatter (Spotless / google-java-format) is the canonical standard — `spotless:apply` fixes drift, CI `spotless:check` fails on it, and Checkstyle enforces the convention rules a formatter does not; for documentation, Checkstyle enforces Javadoc presence on the public API. The right column, "Substantive → human (the judgment a check cannot make)," holds: for review, humans judge design, logic, correctness, and "is this the right change?" on a small pull request of about 100–300 lines; for standards, humans never debate formatting once the formatter is the source of truth; for documentation, humans write the contract (params, `@throws`, pre/post, nullability) and the why (an ADR's decision and consequences). A note spanning the foot states the shared failure mode, identical in all three: automation enforces presence, never quality — a coverage gate confirms tests exist not that they assert anything, a formatter confirms style is consistent not that the design is good, a Javadoc-presence check confirms a comment is there not that it is true, and a required review confirms someone clicked approve not that they read it. Each is a floor, not a ceiling; the "LGTM" on the 2,400-line pull request, the perfectly-formatted God class, and the present-but-stale Javadoc are each presence without substance.
