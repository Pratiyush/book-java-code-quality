Figure 37.2 — One move, three practices: automate the mechanical, reserve humans for the substantive

Review, standards, and documentation each take a category of mechanical work off the human’s plate — so scarce human attention goes only to what only a human can do.

Mechanical → automated the bot does it — cheap, deterministic, at scale
Substantive → human the human judgment a check cannot make

Code review

human catch · Ch 34 bot layer
Bots check Style, lint, and coverage gates run before a human looks — the mechanical findings are already triaged.
→
Humans judge Design, logic, correctness, and “is this the right change?” — on a small PR (~100–300 LOC) the reviewer can actually think about.

Coding standards

Ch 6 formatter · Ch 16 Checkstyle
Formatter applies A deterministic formatter (Spotless / google-java-format) is the canonical standard: spotless:apply fixes drift, CI spotless:check fails on it; Checkstyle enforces the convention rules a formatter does not.
→
Humans never debate it Once the formatter is the source of truth, reviewers never comment on formatting — style is not a choice anymore. Human attention returns to substance.

Documentation

Ch 16 Javadoc-presence check
Checkstyle enforces Javadoc presence on the public API — the comment is required to be there.
→
Humans write The contract (params, @throws, pre/post, nullability) and the why (an ADR’s decision + consequences) that a presence check cannot evaluate.
The shared failure mode — the same in all three: automation enforces presence, never quality, and quality is the human judgment a check cannot replace. A coverage gate confirms tests exist, not that they assert anything; a formatter confirms style is consistent, not that the design is good; a Javadoc-presence check confirms a comment is there, not that it is true; a required review confirms someone clicked approve, not that they read it. Each is a floor, not a ceiling — necessary, automatable, and worthless if mistaken for the quality it gestures at. The “LGTM” on the 2,400-line PR, the perfectly-formatted God class, the present-but-stale Javadoc: each is presence without substance.
