Fig 07.1 — The typography / meaning axis

What a tool can settle once vs. what only a human with domain knowledge can judge — the spine of Chapter 6

Tool-decidable (typography)
Human-only (semantics)
Regex / reflow → one correct answer Domain truth → no correct answer from syntax
Tool checks / rewrites
Human judges
1Naming
Case — typographical, regex-enforceable
lowerCamelCase for methods, fields, params, locals;
UpperCamelCase for types;
CONSTANT_CASE for deeply-immutable static final.
Checkstyle ConstantName ^[A-Z][A-Z0-9]*(_[A-Z0-9]+)*$ Checkstyle MethodName ^[a-z][a-zA-Z0-9]*$ PMD ClassNamingConventions.classPattern [A-Z][a-zA-Z0-9]* SonarQube java:S100 · S101 · S115 · S116 · S117
Meaning — semantic, unjudgeable by regex
data passes every naming rule. It tells no one that it holds a customer's unsettled invoices. Only a person who understands the domain can flag it and rename it outstandingInvoices.
Enforcing naming with a linter does not give good names.
2Structure
File layout + overload contiguity — checkable
Package → imports → one top-level class (Google Java Style §3).
No wildcard imports; static imports in one group; ASCII-sorted within group (§3.3).
Overloaded methods in a single contiguous group — no interleaving (§3.4.2).
A formatter rewrites import order automatically.
Member order — judgment, deliberately rule-free
Google Java Style §3.4.2 explicitly declines a single recipe: "what is important is that each class uses some logical order, which its maintainer could explain if asked."
Gating member order mechanically (beyond overload contiguity) fights the guide's own intent.
3Formatting
Fully decidable — formatter produces the answer
Google Java Style §4: indentation +2 spaces (§4.2), 100-char column limit (§4.4), K&R braces (§4.1), one statement per line (§4.3).
google-java-format 1.35.0 — no configurability by design palantir-java-format — 120-char, chain-tuned Spotless 3.6.0 — spotless:check / spotless:apply EditorConfig — indent_style, indent_size, end_of_line, charset
Style value choices — pick one; all are defensible
Google uses 2-space / 100-col; AOSP uses 4-space; palantir-java-format uses 120-col; Checkstyle LineLength defaults to 80. No value is correct; the value is in picking one and stopping the argument. Format/lint split: once a formatter owns whitespace, delete the overlapping linter whitespace rules (Ch 17).
Enforced by tool
  • Formatter rewrites the file (no human fix needed)
  • Linter checks case via regex; flags deviations
  • CI gate: spotless:check fails the build on whitespace drift
  • -Xdoclint all,-missing shape-checks Javadoc without forcing one on every member
Resolved by human reviewer
  • Is the name true? Does it carry the domain concept?
  • Is the member order explainable? Could its maintainer give a reason?
  • Does the comment capture the why the code cannot?
  • Has the threshold between "rename first" and "comment the why" been found?
Format/lint split (CONCEPT). A linter detects a whitespace deviation and asks a human to fix it; a formatter computes the canonical rendering and writes it. Formatting is fully decidable, so it belongs to the formatter. Once it does, the whitespace-checking linter rules are deleted — two tools must not fight over the same bytes.
Convention vs meaning (CONCEPT). The settled-feeling parts of "readability" are exactly the typographical ones — which is why they are safe to automate, and why automating them reveals nothing about whether names are semantically true.
