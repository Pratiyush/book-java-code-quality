Fig 14.2 — Type Erasure: Compile Time vs. Run Time

Generic type information is a compile-time artifact. The JVM sees only raw types and the casts the compiler inserted. (JLS SE 21 §4.6)

JLS §4.6 erasure rules — what the compiler maps each generic construct to:
Param. type List<String> ⟶ erases to ⟶ List  (raw form)
Type variable <T> ⟶ erases to ⟶ Object  (leftmost bound; unbounded T → Object)
Bounded var <T extends Number> ⟶ erases to ⟶ Number  (leftmost bound; JLS §4.6)
Compile time
javac · -Xlint:unchecked,rawtypes · type arguments visible
✔
Type-argument checking
Compiler verifies every read/write against the declared type argument. An Integer cannot be added to a List<String>.
✔
Cast insertion
Compiler inserts a checked cast at every point where a type argument is used: strings.get(0) → (String) list.get(0). Correct generics mean casts never fail.
!
Unchecked warning (JLS §5.1.9)
Emitted on any raw-type assignment or unchecked cast. "The compiler has done all it can; erasure leaves a gap it cannot close." Surfaced by -Xlint:unchecked,rawtypes and Sonar java:S3740.
✔
Wildcard & bound checking (JLS §4.10.2)
Compiler enforces: ? extends T → read-only; ? super T → write-only. PECS correctness is enforced here.
✔
Type erasure (performed here)
All type arguments are stripped from the bytecode. List<String> becomes List in the .class file. §4.6.

Run time (JVM)
HotSpot · bytecode · type arguments absent
✘
No type argument — raw type only
List<String> and List<Integer> are the same class: List. The JVM cannot distinguish them. JLS §4.6.
✔
Inserted casts execute
The casts the compiler inserted run. If the source code was type-safe (no unchecked warnings), they all succeed. If a raw-type smuggle occurred, one of these casts throws ClassCastException — at the get, far from the offending put.
✘
Non-reifiable types cannot be checked (JLS §4.7)
instanceof List<String> is illegal. Only instanceof List<?> or bare instanceof List is permitted. Generic arrays (new List<String>[]) are illegal for the same reason.
!
Heap pollution (JLS §4.12.2)
A variable of a parameterized type holds an object of a different type. Arises from raw-type assignments or unsafe varargs. The JVM detects it only when the compiler-inserted cast runs.
✘
Generic array creation forbidden
new T[n], new List<String>[] are compile errors: an array of a non-reifiable type cannot uphold the array contract (JLS §4.7; Item 28).
Reifiable (JLS §4.7) — JVM can check Non-generic types · primitives · raw types · List<?> (unbounded wildcard) · arrays of reifiable types
Non-reifiable — JVM cannot check List<String> · List<? extends Number> · any concrete type argument. Result: instanceof forbidden; generic arrays forbidden.
The unchecked warning is the compiler's debt notice. It means: "I have done all I can; erasure leaves a gap I cannot close; a runtime cast failure is possible." Item 27 (Effective Java 3e): eliminate every unchecked warning. If elimination is impossible, suppress with @SuppressWarnings("unchecked") at the narrowest scope and add a comment proving the operation is type-safe. The annotation discharges the debt; the comment is the proof.

Sources: JLS SE 21 §4.6 (type erasure rules, parameterized→raw, type-var→leftmost-bound); §4.7 (reifiable / non-reifiable types); §4.12.2 (heap pollution); §5.1.9 (unchecked conversion); Effective Java 3e Items 26–28 (raw types, unchecked warnings, lists vs arrays); compiler flag -Xlint:unchecked,rawtypes; Sonar rule java:S3740 (raw types).
