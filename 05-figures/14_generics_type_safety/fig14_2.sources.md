# Fig 14.2 Sources — Type Erasure: Compile Time vs. Run Time
Chapter 11 (dossier key 14): Let the Compiler Carry It

| Label / atom in diagram | Claim | Pinned source (SOURCE-PIN.md) |
|---|---|---|
| "Generic type information is a compile-time artifact" | Type erasure (JLS §4.6): the compiler maps a parameterized type to its raw form; generic information is checked, used to insert casts, then thrown away | JLS SE 21 §4.6 (type erasure) |
| Erasure rule: `List<String>` → `List` (raw form) | A parameterized type erases to its raw type | JLS SE 21 §4.6, first transformation rule |
| Erasure rule: `<T>` → `Object` (unbounded T → Object) | A type variable with no bound erases to `Object` (its leftmost bound) | JLS SE 21 §4.6, second transformation rule |
| Erasure rule: `<T extends Number>` → `Number` (leftmost bound) | A bounded type variable erases to the erasure of its leftmost bound | JLS SE 21 §4.6, second transformation rule |
| "Compiler verifies every read/write against the declared type argument" | Type-argument checking at compile time: the compiler catches an `Integer` being added to a `List<String>` | JLS SE 21 §4.5 (parameterized types); §4.6 preamble |
| "Cast insertion — compiler inserts a checked cast at every point where a type argument is used" | Compiler inserts synthetic casts when erasing generics, e.g. `strings.get(0)` becomes `(String) list.get(0)` | JLS SE 21 §4.6 (cast insertion is the mechanism that preserves correctness after erasure) |
| "Unchecked warning (JLS §5.1.9)" | An unchecked warning is emitted on any unchecked conversion (raw-type assignment); "the compiler has done all it can" | JLS SE 21 §5.1.9 (unchecked conversion); Effective Java 3e Item 27 |
| `-Xlint:unchecked,rawtypes` | Compiler flag that surfaces every unchecked and raw-type warning | Draft back-matter §Tool rules; cited in draft §"Raw types and the unchecked warning" |
| "Sonar java:S3740" | Sonar rule flagging raw types | Sonar rule java:S3740 (cited in draft §How it works) |
| "`List<String>` and `List<Integer>` are the same class: `List` at run time" | JVM cannot distinguish parameterized types after erasure; both are just `List` | JLS SE 21 §4.6 |
| "Inserted casts execute — ClassCastException at the get, far from the offending put" | Heap-pollution scenario from draft hook: `raw.add(42)` smuggles Integer; the cast at `strings.get(0)` throws ClassCastException | JLS SE 21 §4.6 (cast insertion); draft hook example §1 |
| "Non-reifiable types cannot be checked (JLS §4.7)" | `instanceof List<String>` is illegal; only `instanceof List<?>` or bare `instanceof List` is allowed | JLS SE 21 §4.7 (reifiable types — verbatim list) |
| Reifiable list: non-generic types, primitives, raw types, `List<?>`, arrays of reifiable | Verbatim reifiable types per JLS §4.7 | JLS SE 21 §4.7 |
| Non-reifiable list: `List<String>`, `List<? extends Number>` | Concrete type arguments are non-reifiable; result: instanceof and generic-array creation forbidden | JLS SE 21 §4.7 |
| "Generic array creation forbidden — `new T[n]`, `new List<String>[]`" | A generic array creation is a compile error because an array of a non-reifiable type cannot uphold the array store contract | JLS SE 21 §4.7; Effective Java 3e Item 28 ("prefer lists to arrays") |
| "Heap pollution (JLS §4.12.2)" | A variable of a parameterized type holds an object of a different type; arises from raw-type assignments or unsafe varargs | JLS SE 21 §4.12.2 (heap pollution) |
| "`@SuppressWarnings("unchecked")` at narrowest scope with proof comment" | Item 27: when elimination is impossible, suppress at narrowest scope and prove safety in a comment | Effective Java 3e Item 27 (Bloch, 2018) |
| "Wildcard & bound checking (JLS §4.10.2)" | Compiler enforces `? extends T` read-only / `? super T` write-only at compile time | JLS SE 21 §4.10.2 |
