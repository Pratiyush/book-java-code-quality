## Accessibility

- Alt-text: Type-erasure timeline: compile-time checks and cast insertion erase to a run-time JVM that sees only raw types and casts.
- Long-description: A diagram titled "Fig 11.2 — Type Erasure: Compile Time vs. Run Time," subtitled that generic type information is a compile-time artifact and the JVM sees only raw types and the casts the compiler inserted (JLS SE 21 §4.6). A strip at the top, "JLS §4.6 erasure rules," gives three mappings: a parameterized type List<String> erases to List (raw form); a type variable <T> erases to Object (leftmost bound; unbounded T to Object); a bounded variable <T extends Number> erases to Number (leftmost bound). Below the strip, a two-phase timeline runs left to right with a downward arrow labelled "erasure" between the phases. The left phase, "Compile time (javac · -Xlint:unchecked,rawtypes · type arguments visible)," lists five facts: type-argument checking, marked allowed (the compiler verifies every read/write against the declared type argument; an Integer cannot be added to a List<String>); cast insertion, marked allowed (the compiler inserts a checked cast wherever a type argument is used, so strings.get(0) becomes (String) list.get(0), and correct generics mean casts never fail); unchecked warning (JLS §5.1.9), marked caution (emitted on any raw-type assignment or unchecked cast, surfaced by -Xlint:unchecked,rawtypes and Sonar java:S3740); wildcard and bound checking (JLS §4.10.2), marked allowed (? extends T read-only, ? super T write-only, PECS enforced here); and type erasure performed here, marked allowed (all type arguments are stripped from the bytecode; List<String> becomes List in the .class file). The right phase, "Run time (JVM) (HotSpot · bytecode · type arguments absent)," lists five facts: no type argument, raw type only, marked not-possible (List<String> and List<Integer> are the same class, List, and the JVM cannot distinguish them); inserted casts execute, marked allowed (the compiler-inserted casts run; if the source was type-safe they all succeed, but a raw-type smuggle makes one throw ClassCastException at the get, far from the offending put); non-reifiable types cannot be checked (JLS §4.7), marked not-possible (instanceof List<String> is illegal, only instanceof List<?> or bare instanceof List is allowed, and generic arrays are illegal too); heap pollution (JLS §4.12.2), marked caution (a variable of a parameterized type holds an object of a different type, arising from raw-type assignments or unsafe varargs, detected only when the inserted cast runs); and generic array creation forbidden, marked not-possible (new T[n] and new List<String>[] are compile errors). Below the timeline, a two-box reference strip contrasts reifiable types the JVM can check (non-generic types, primitives, raw types, List<? > unbounded wildcard, arrays of reifiable types) against non-reifiable types it cannot check (List<String>, List<? extends Number>, any concrete type argument — so instanceof and generic arrays are forbidden). A closing box states the unchecked warning is the compiler's debt notice meaning a runtime cast failure is possible, and Item 27 of Effective Java 3e: eliminate every unchecked warning, or if elimination is impossible suppress it with @SuppressWarnings("unchecked") at the narrowest scope plus a comment proving type safety. A source line closes the figure.
