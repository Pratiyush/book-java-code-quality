# Fig 11.1 Sources — PECS Variance Ladder
Chapter 11 (dossier key 14): Let the Compiler Carry It

| Label / atom in diagram | Claim | Pinned source (SOURCE-PIN.md) |
|---|---|---|
| "Java generics are invariant" | `List<String>` is not a subtype of `List<Object>` even though `String extends Object` | JLS SE 21 §4.10.2 (subtyping of parameterized types — variance) |
| "`List<String>` is not a `List<Object>`" | Invariance: assigning a `List<String>` to a `List<Object>` variable is a compile error | JLS SE 21 §4.10.2 |
| "? extends T — covariant, Producer-Extends" | Wildcard `? extends T` makes a parameterized type covariant; the element type is an unknown subtype of T, so only reads (returning T) are safe; writes are forbidden (except null) | JLS SE 21 §4.10.2; Effective Java 3e Item 31 (Bloch, 2018) |
| "? super T — contravariant, Consumer-Super" | Wildcard `? super T` makes the parameterized type contravariant for the lower bound; adds of T or subtypes are safe; reads come back only as Object | JLS SE 21 §4.10.2; Effective Java 3e Item 31 |
| "PECS — Producer-Extends, Consumer-Super" | The PECS mnemonic: if a parameter produces T values use `? extends T`; if it consumes T values use `? super T` | Effective Java 3e Item 31 (Bloch, 2018) |
| `static <T> void copy(List<? extends T> src, List<? super T> dst)` | The canonical copy idiom combining producer-extends and consumer-super from Item 31 | Effective Java 3e Item 31 |
| "Unbounded wildcard `<?>`" | `List<?>` — element type is irrelevant; only type-independent operations (iteration, size) are safe | JLS SE 21 §4.5.1 (wildcard type arguments) |
| "never use a bounded wildcard as a return type" | A wildcard return type forces every caller to propagate the wildcard outward, leaking complexity; use a concrete type in return position | Effective Java 3e Item 31 ("Do not use wildcard types as return types") |
| "Sonar java:S1452" | Sonar rule flagging wildcard return types | Sonar rule java:S1452 (cited in draft §Variance as contested for interface-mandated wildcard returns) |
| "? extends Number accepts List<Integer> or List<Double>" | A `List<Integer>` and `List<Double>` both satisfy `List<? extends Number>` — covariant widening | JLS SE 21 §4.10.2 |
| "? super Integer accepts List<Number> or List<Object>" | A `List<Number>` and `List<Object>` both satisfy `List<? super Integer>` — contravariant widening | JLS SE 21 §4.10.2 |
| "cannot add (except null) to ? extends T" | Writes to an upper-bounded wildcard are forbidden because the actual type argument is unknown (could be any subtype of T); only null is permitted | JLS SE 21 §4.10.2; Effective Java 3e Item 31 |
| "reads come back only as Object from ? super T" | A lower-bounded wildcard: the actual type is unknown upward, so reads can only be typed as Object | JLS SE 21 §4.10.2; Effective Java 3e Item 31 |
