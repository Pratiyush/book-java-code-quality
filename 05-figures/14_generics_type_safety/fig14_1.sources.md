| "? extends T — covariant, Producer-Extends" | Wildcard `? extends T` makes a parameterized type covariant; the element type is an unknown subtype of T, so only reads (returning T) are safe; writes are forbidden (except null) | JLS SE 21 §4.10.2; Effective Java 3e Item 31 (Bloch, 2018) |

| "Unbounded wildcard `<?>`" | `List<?>` — element type is irrelevant; only type-independent operations (iteration, size) are safe | JLS SE 21 §4.5.1 (wildcard type arguments) |
| "never use a bounded wildcard as a return type" | A wildcard return type forces every caller to propagate the wildcard outward, leaking complexity; use a concrete type in return position | Effective Java 3e Item 31 ("Do not use wildcard types as return types") |

## Accessibility

- Alt-text: PECS variance ladder: three cards — invariant List<String>, covariant ? extends T (producer), contravariant ? super T (consumer).
- Long-description: A diagram titled "Fig 11.1 — PECS Variance Ladder," subtitled that Java generics are invariant at declaration and wildcards restore controlled flexibility at the use site. A banner across the top reads "PECS — Producer-Extends · Consumer-Super — Item 31, Effective Java 3e (Bloch, 2018)," and explains that if a parameter produces T values use ? extends T, and if it consumes T values use ? super T. Below the banner sit three cards left to right. The left card, "Declaration-site · invariant — Exact type," signature List<String>, lists operations: read as String (allowed), add String (allowed), assign to List<Object> (not allowed — invariance, not a subtype, JLS §4.10.2); its note states List<String> is not a List<Object> even though String extends Object, and invariance keeps the type system sound. The middle card, "Use-site · covariant (producer) — ? extends T," tagged "PECS — Producer-Extends," signature List<? extends Number>, lists: read as Number, the upper bound (allowed); add anything except null (not allowed — actual type unknown); accept List<Integer> or List<Double> (allowed); its note states the parameter produces T values for the caller to read and widens what callers can pass in. The right card, "Use-site · contravariant (consumer) — ? super T," tagged "PECS — Consumer-Super," signature List<? super Integer>, lists: add Integer or a subtype (allowed); read as anything more specific than Object (not allowed); accept List<Number> or List<Object> (allowed); its note states the parameter consumes T values from the caller and widens what callers can pass in. Below the cards, a two-pill strip: "Both at once (copy idiom)" showing static <T> void copy(List<? extends T> src, List<? super T> dst) — src produces T (extends), dst consumes T (super), Item 31 — and "Unbounded wildcard <?>" showing static void printAll(List<?> list) — the element type is irrelevant, only iteration/size operations are needed, JLS §4.5.1. A closing hard-rule box states: never use a bounded wildcard as a return type — a method returning List<? extends Number> forces every caller to deal with the wildcard, so use a concrete type in return position, and Sonar java:S1452 flags wildcard return types (Item 31). A source line closes the figure.
- Grayscale/contrast/legibility note: Grayscale-safe.
