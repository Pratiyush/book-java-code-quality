**Figure:** Fig 10.2 — try-with-resources: suppressed vs masked exceptions  
**Chapter:** 10 (dossier key 12) — When Things Go Wrong  
**Family:** Flow-sequence (two-column contrast)  
**One idea:** try-with-resources (JLS §14.20.3) closes resources in LIFO order on every path and suppresses a close() exception via Throwable.addSuppressed rather than masking the original — the exact silent bug the old try/finally idiom commits (Effective Java Item 9).

## Accessibility

- Alt-text: Side-by-side: old try/finally masks failure E1 (left) versus try-with-resources suppressing E2 and preserving E1 (right).
- Long-description: A diagram titled "Fig 10.2 — try-with-resources: suppressed vs masked exceptions," subtitled "JLS SE 21 §14.20.3 · Effective Java 3e Item 9 · Throwable.addSuppressed/getSuppressed (JDK 21 API)." Two panels sit side by side. The left panel, headed "Old pattern: try/finally" with the badge "BUG: masks original exception," lists four steps: (1) the body executes and throws E1, the real failure (read error on the stream); (2) finally { resource.close(); } runs and close() throws E2; (3) E2 propagates and E1 is silently discarded, so the real failure and its stack trace are gone; and a final step marked with a cross noting that with multiple resources and nested finally each additional close can mask the previous exception, quoting Effective Java Item 9, "The code is even more inscrutable." Its outcome box reads "E1 (root cause) lost forever — no way to recover or even log the original failure." The right panel, headed "try-with-resources" with the badge "CORRECT: suppresses, never masks," lists four steps: (1) the body executes and throws E1, the real failure; (2) resources close in reverse order, LIFO, per JLS §14.20.3 — last opened closes first, each close runs on every path; (3) if close() throws E2, E1.addSuppressed(E2) is called automatically, E1 propagates, and E2 is attached, not lost; and a final step marked with a check noting that E2 is retrieved via e.getSuppressed(), still accessible for diagnostics. Its outcome box reads "E1 propagates · E2 attached as suppressed — root cause preserved, caller can inspect both, JDK 21 API: Throwable.getSuppressed()." Below the two panels, a strip of three cards: "Close order" (LIFO, reverse of declaration; in try (A a = …; B b = …), b.close() runs before a.close(); JLS §14.20.3); "Every path" (close() runs on normal completion, on a body throw, and even if a later resource's initialization fails; JLS §14.20.3); and "AutoCloseable contract" (AutoCloseable.close() is not required idempotent, Closeable, its subtype, is required idempotent, write custom implementations idempotent anyway; JDK 21 API). A linter row below states that resource leaks trigger Sonar java:S2095, PMD CloseResource, SpotBugs OS_OPEN_STREAM / OBL_UNSATISFIED_OBLIGATION, and Error Prone MustBeClosed / StreamResourceLeak, which fire when a resource escapes the try-with-resources header (created in the body, returned from a factory, or stored in a field). A source line closes the figure.
