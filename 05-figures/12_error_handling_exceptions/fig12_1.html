Effective Java 3e, Item 70 — decision rule (one sentence per kind)
Checked (Exception − RuntimeException)
Use for conditions the caller can reasonably recover from — e.g. a file that may not exist, a remote call that may time out.
Enforced by: Checkstyle IllegalThrows · Sonar java:S112
Unchecked (RuntimeException)
Use for programming errors — broken preconditions. "When in doubt, throw unchecked" (Item 70).
Enforced by: PMD AvoidThrowingRawExceptionTypes · Sonar java:S112
Error
Reserved for the JVM (resource exhaustion, unrecoverable state). Do not throw or subclass — except AssertionError.
Item 70: "you shouldn't throw them either, and you should not subclass them."
The failure model is a contract. The checked exceptions a module declares are API: callers plan recovery around them. Unchecked exceptions signal bugs callers are expected to debug. Choosing the wrong kind — or swallowing one — breaks the contract as surely as returning the wrong value. Catching too broadly (catch (Exception e) {}) violates Items 70 + 77 and is flagged by Checkstyle IllegalCatch, PMD EmptyCatchBlock, and SpotBugs DE_MIGHT_IGNORE.
