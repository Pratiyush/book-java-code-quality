## Accessibility
- Alt-text: A four-stage CI gate ordered by feedback latency — pre-commit, PR-fast, main/nightly, merge — with un-bypassable enforcement at the merge. (118 chars)
- Long-description: Four gate stages run left to right, joined by right-pointing arrows, each card naming its latency, where it runs, the checks it carries, and its outcome. Stage 0, "Pre-commit" (seconds, on the developer's machine; marked as the start with a left accent bar), runs Spotless format (Ch 6) and a gitleaks secrets scan (Ch 31, 35); outcome: pass proceeds to commit, fail is blocked locally. Stage 1, "PR-fast" (a few minutes, in CI on every pull request), runs compile plus Error Prone (Ch 18), Checkstyle with a curated ruleset (Ch 16), unit tests with JUnit 6 (Part V), and JaCoCo coverage on new code only (Ch 23, 34); outcome: fail on a new high-severity finding blocks the PR. Stage 2, "Main / nightly" (slow, off the fast path, in CI on a schedule or post-merge), runs SpotBugs 4.10.2 plus FindSecBugs (Ch 16), SonarQube full analysis (Ch 17, 33), OWASP Dependency-Check SCA (Ch 28), PITest mutation on critical modules (Ch 23), and integration tests with Testcontainers (Ch 21); outcome: findings go to team review and do not block individual PRs. Stage 3, "Merge (enforcement)" (using the PR-fast result; gated by branch protection plus a merge queue; shown with a darker header to mark it as the enforcement point), requires the PR-fast result as a required status check (Ch 35), branch protection on main (Ch 35), and a merge queue that prevents bypass (Ch 35); outcome: a gate not passed blocks the merge, a gate passed keeps main green. Below the four stages, four design-principle chips read: fast feedback first (cheap checks — format, compile, unit — run pre-commit and on every PR so developers never wait for the slow stage); new-code focus or clean-as-you-code (the PR gate blocks on the quality of what the team touches, making it adoptable on legacy without a wall of existing findings); un-bypassable enforcement (required status check plus branch protection plus merge queue means no path to main skips the gate); and slow checks off the fast path (SpotBugs, Sonar, SCA, and mutation run nightly for full rigor where it is affordable). A closing note explains why four stages rather than one: a single gate running everything on every PR would be too slow to sustain, while one that skips the slow checks would miss the deep findings, so the feedback-latency ladder gives developers instant signal from fast checks, the team thorough signal from slow checks on a schedule, and locks enforcement at the merge so neither is optional — and the new-code focus is what makes the design adoptable on a legacy codebase with thousands of existing findings.
