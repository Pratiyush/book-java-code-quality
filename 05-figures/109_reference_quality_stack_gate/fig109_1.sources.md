## Accessibility
- Alt-text: A nine-layer reference Java quality stack — build through platform — each layer naming a primary tool, an equally-valid alternative, and its cost. (122 chars)
- Long-description: A layered stack laid out as a table of nine rows under four column headers: concern layer, primary tool(s) and what they catch, named alternative and when to swap, and cost/limit. Read top to bottom: (1) Build — Maven 3.9.16 plus wrapper (reproducible build, pinned plugin versions); alternative Gradle 9.6.0 (more flexible, more complex; swap when the ecosystem or build logic favors it); cost is XML verbosity and the wrapper requirement. (2) Format — Spotless 3.6.0 plus google-java-format 1.35.0 (style consistency auto-fixed, ending the style debate and taking it off review); alternative palantir-java-format (an equal house style; the choice matters less than automating one); cost is one imposed, non-negotiable style. (3) Style and convention — Checkstyle 13.6.0 with a curated ruleset (naming, imports, Javadoc presence, conventions a formatter does not reach); alternative PMD 7.25.0 alongside as a curated subset; cost is noise if over-configured. (4) Bug-finding, the two-layer row set off by an accent bar — Error Prone 2.x at compile time (fast in-compiler feedback on bug patterns, source-level) and SpotBugs 4.10.2 plus FindSecBugs at the bytecode level (a different bug class plus security findings); the alternative is either alone if build time is tight, but the duo is deliberate because they see different things; cost is build time, false positives, and two configs to own. (5) Null-safety — NullAway 0.13.4 plus JSpecify 1.0.0 (NullPointerException caught at build time, annotations making intent explicit); alternative Checker Framework 4.2.0 (stronger guarantees at higher annotation cost); cost is annotation effort and that it is not a total guarantee. (6) Architecture — ArchUnit 1.4.2 run as tests (dependency cycles and layering violations enforced as ordinary JUnit tests); alternative Sonar architecture rules or manual review for smaller codebases; cost is that rules must be written and maintained. (7) Tests and coverage — JUnit 6.1.0 plus AssertJ 3.27.7 plus Mockito 5.23.0 plus Testcontainers 2.0.5 for correctness, with JaCoCo 0.8.15 coverage and PITest 1.25.3 mutation on critical modules; the alternative is to vary mock/assertion libraries to taste; cost is that mutation testing is slow and is reserved for critical paths. (8) Security and supply-chain — OWASP Dependency-Check 12.2.2 (SCA) plus gitleaks (secrets) plus CycloneDX 1.6 (SBOM), with Semgrep 1.163.0 or CodeQL (SAST) where warranted; alternative commercial SCA/SAST platforms (both OSS and commercial are legitimate); cost is SCA false positives and SAST tuning effort. (9) Platform — SonarQube Server 2026.1 LTA with a new-code, clean-as-you-code gate (aggregates findings, trends, and a single gate verdict on new code; the Community edition is OSS); alternative a CI-native gate composing the tools without a platform; cost is a server to run and own. A note states this is one defensible setup, not the setup — every pick names its trade-off and an equally-valid alternative, an all-OSS core is possible (SonarQube Community plus OSS analyzers), the full stack is code to own, and adoption should be incremental (format first, then fast linters, then heavier analysis), because tools are scaffolding while design, review, and culture are where quality is decided (Ch 1, 37).
