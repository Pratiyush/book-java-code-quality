## Accessibility

- Alt-text: Fowler's technical-debt 2x2 matrix: columns reckless vs prudent, rows deliberate vs inadvertent, each cell with a quote and a verdict.
- Long-description: A two-by-two matrix. The columns are labelled "Reckless" (left) and "Prudent" (right); the rows are labelled "Deliberate" (top) and "Inadvertent" (bottom). Each of the four cells holds a quoted sentence a team says, a one-line illustrative Java comment, and a verdict badge reading either "Damage" or "Tool". Top-left, Deliberate and Reckless: "We don't have time for design.", comment "2 000-line class — just ship it; no design, no test, no review", verdict Damage ("debt with no plan to repay"). Top-right, Deliberate and Prudent: "We must ship now and will deal with the consequences — knowingly.", comment "TODO(#472): extract validator — tracked; ships 2024-03-01; planned sprint 24-07", verdict Tool ("tracked, time-boxed, repaid"). Bottom-left, Inadvertent and Reckless: "What's layering?", comment "UI class calling DB directly; developer unaware of layering", verdict Damage ("ignorance, not intent"). Bottom-right, Inadvertent and Prudent: "Now we know how we should have done it.", comment "initial design used inheritance; composition is clearly better now", verdict Tool ("refactor as knowledge grows"). A legend states that prudent cells are a tool (debt tracked or repaid as understanding grows) and reckless cells are damage (no plan, no visibility, interest accrues silently). A note attributes the original metaphor to Ward Cunningham (OOPSLA 1992) and the quadrant to Fowler's TechnicalDebt bliki, closing with the book's line that prudent tracked debt is a tool and reckless debt is not.
- Grayscale-safe: Yes. The reckless and prudent cells differ slightly in fill, but the load-bearing distinction is carried by the word labels "Damage" / "Tool" on each badge and by the row/column headers, not by hue. Reads in black-and-white print.
- Contrast: Sufficient. Dark text on light cell fills; the quoted lines and verdict text are legible at print size.
- Code legibility: The one-line Java comments are real authored text (not a screenshot), set in monospace, and readable without syntax color; they are illustrative comments, so they carry no syntax-highlighted tokens to lose.
