## Accessibility

- Alt-text: The test pyramid — many unit tests at the wide base, fewer integration tests, fewest end-to-end tests — beside its inverted ice-cream-cone anti-pattern.

- Long-description: The figure is titled "Fig 20.1 — The test pyramid," subtitled with two Vocke/Cohn rules quoted verbatim: "Write tests with different granularity" and "The more high-level you get the fewer tests you should have." The left column holds the pyramid, drawn as three stacked horizontal blocks that widen from top to bottom, each block lighter as it widens. The narrow top block is UI / End-to-End — full-stack behaviour, slowest and costliest to run and maintain — badged "fewest." The middle, wider block is Integration / Service — the application "correctly interacts with external dependencies like databases, filesystems, and separate services" — badged "fewer." The wide base block is Unit — "focused, isolated tests of individual code components with the narrowest scope," fast, cheap, plentiful — badged "most." A note under the pyramid states the build expression: Maven Surefire runs unit tests in the test phase; Maven Failsafe runs *IT integration tests in the verify phase; that split is the pyramid in the build. Below the pyramid, a bordered panel titled "Anti-pattern: the ice-cream cone" shows the same three layers inverted — Unit (few) narrow at the top, Integration in the middle, and E2E (many, widest base) at the bottom — captioned with Vocke's verbatim warning that excessive high-level tests "become maintenance nightmares and run slowly" and that E2E tests are "notoriously flaky and often fail for unexpected and unforeseeable reasons."

  The right column is a stack of note cards. Two cards quote Vocke rule 1 and rule 2 verbatim (as in the subtitle). A "Heuristic, not a law" card notes that Vocke's layer names are "deliberately fuzzy," the shape is a rule of thumb rather than a 70/20/10 quota, and alternative weightings (the testing trophy, the honeycomb) put more weight on integration tests — each a different trade-off, none crowned. A "Solitary vs. sociable" card distinguishes solitary unit tests (stub all collaborators for isolation) from sociable ones (let the unit interact with real collaborators where practical), noting this is the root of the mocking debate routed to Chapter 21. A further card quotes Vocke's prescription verbatim: "Test observable behaviour, not implementation details." A final "Deep routes" card lists where each tool gets depth: Ch 21 (JUnit 5, assertions, Mockito), Ch 22 (Testcontainers, integration layer), and later chapters (PITest mutation and JaCoCo gating, jqwik property-based, REST-assured / Pact at the API seam).
