## Accessibility

- Alt-text: A 2x2 grid of coverage versus mutation score, with the high-coverage low-score "vanity suite" corner flagged as where coverage lies.

- Long-description: The figure is titled "Fig 20.2 — Coverage vs. mutation score: two independent axes of test quality," subtitled: coverage measures execution, mutation score measures fault detection, they are independent, and a suite can reach 100% coverage and kill zero mutants. It has three columns.

  The left column defines the two axes. The X-axis card, Coverage (execution): JaCoCo reports six counters verbatim — Instructions, Branches, Lines, Methods, Classes, Cyclomatic Complexity — measuring which instructions ran while tests executed, with branch coverage (every if / switch arm) the stronger floor; PITest verbatim: coverage "measures only which code is executed by your tests." The Y-axis card, Mutation score (detection): PITest "automatically seeds" small faults then reruns the suite per mutant; a killed mutant means a test failed (fault detected), a lived mutant means all tests passed (suite blind to that fault); score = killed ÷ (killed + lived) × 100%. Two quoted callouts follow: PITest verbatim, "It does not check that your tests are actually able to detect faults in the executed code," and the chapter thesis that coverage is necessary, not sufficient — a floor that finds untested code, never proof that tested code is well tested.

  The centre column is a 2×2 matrix. Its X-axis runs coverage low (left) to high (right); its Y-axis runs mutation score high (top) to low (bottom). The four quadrants: top-right, High coverage and high score — tests execute the code and detect faults; strong boundary @ParameterizedTest plus AssertJ assertions that verify the result — badged "target state." Top-left, Low coverage, high score — the tests that run cover their code well but overall coverage is low; expand coverage — badged "improve coverage." Bottom-right, High coverage, low score — drawn with a heavier red border and marked "▲ chapter hook" — every line runs yet tests assert almost nothing; the coverage badge is green while the mutation gate is red; the "vanity suite" of 100% line coverage and 0% kill rate, where a real money-losing bug passes uncaught — badged "coverage lies here." Bottom-left, Low coverage, low score — most code never runs and tested code has weak assertions; start here, adding coverage and strengthening assertions together — badged "both axes weak." A callout strip under the matrix, "The chapter hook: vanity suite," states that a DiscountPolicy test suite reaches 100% line coverage yet kills under 30% of seeded mutants because a >= → > conditional mutation survives — the tests call the method but never assert the returned discount amount; coverage gate green, mutation gate red, the suite lies.

  The right column gives tool detail. A JaCoCo block lists the six counters again as a bulleted list and notes the check goal has no built-in default threshold (verify at pin). A PITest block shows the three-step mutation lifecycle — (1) seed a fault automatically (flip a conditional, change + to -, replace a return value), (2) rerun the suite against the mutant, (3) outcome: killed = test failed (fault detected) or lived = all passed (suite blind) — and notes a full-project run is slow, so scope it to changed code. A FIRST block expands the acronym Fast, Isolated, Repeatable, Self-validating, Timely, naming it a community checklist (not a formal standard) that complements the quantitative axes.
