## Accessibility

- Alt-text: Triage tree mapping each finding to one of four scope levers — suppress one site, tune one rule, baseline, ratchet — ordered narrow to broad.

- Long-description: The figure has two side-by-side halves under the title "Fig 19.1 — Finding triage and the four-lever scope ladder," plus a tool strip beneath.

  The left half, "Step 1 — triage the finding," is a decision tree. An entry node, "A static-analysis finding arrives — what is it, exactly?", branches down to four decision cards: (1) Real defect — tool is right, the code is wrong → Fix it; (2) False positive — tool is wrong at this site, or real but accepted → Lever 1; (3) Rule misfires everywhere — the same wrong pattern across the whole project → Lever 2; (4) Pre-existing legacy debt — too large to fix now, freeze past and guard future → Levers 3 + 4. A callout under the cards states the governing discipline: "Breadth is always a smell" — reaching for a broad lever (a class-wide @SuppressWarnings, a rule-blind //NOSONAR) when only one site or one rule is wrong silences future real findings; always reach for the narrowest lever that solves the actual problem.

  The right half, "Step 2 — apply the right lever (narrow → broad)," is a four-rung ladder of cards joined by downward arrows, each rung shaded progressively darker to show widening scope. Lever 1, Per-finding suppression (scope: one site): SpotBugs @SuppressFBWarnings(value, justification); PMD @SuppressWarnings("PMD.RuleName") or // NOPMD with a reason; Checkstyle @SuppressWarnings("checkname") requiring SuppressWarningsFilter plus SuppressWarningsHolder on the TreeWalker; Error Prone @SuppressWarnings("CheckName"); NullAway castToNonNull(x) at the exact site; Sonar //NOSONAR (rule-blind — prefer a reviewed False Positive / Accepted transition). Use when the tool is wrong at exactly one site; danger: the broad form silences all future findings of that kind in scope. The arrow to Lever 2 reads "if the same rule misfires everywhere in the project." Lever 2, Rule / ruleset tuning (scope: one rule, project-wide): Checkstyle drop the module or set severity; PMD <exclude name="RuleName"/> or violationSuppressRegex / violationSuppressXPath; SpotBugs a FindBugsFilter Match with no narrowing; Error Prone -Xep:CheckName:OFF (last flag wins); Sonar deactivate the rule in the Quality Profile. Use when one rule misfires the same way project-wide; danger: a global OFF removes signal where the rule would catch a real defect. The arrow to Lever 3 reads "if legacy debt is too large to fix now." Lever 3, Baseline — freeze existing findings (scope: all current findings): SpotBugs baselineFiles ("bugs found in the baseline files won't be reported"; multi-file since spotbugs-maven-plugin 4.7.1.0); Checkstyle external suppressions.xml via SuppressionFilter, with maxAllowedViolations capping by count; Sonar anything outside the New Code window is tracked but excluded from the default gate. Use when a legacy codebase is too noisy to fix on day one — gate the delta, not the total; danger: it freezes real bugs hiding among pre-existing findings. The arrow to Lever 4 reads "combine with lever 3: let past persist, block new." Lever 4 (drawn with a heavier border), Ratchet — "clean as you code" (scope: future code only): Sonar's New Code Definition applies the gate only to new code so legacy does not block the build and debt decays as files are touched; Checkstyle maxAllowedViolations set to today's count then lowered = a count ratchet; SpotBugs regenerating the baseline each release plays the same role. Use when a baseline is already in place; danger: it never forces legacy remediation, count ratchets can be gamed by splitting or reformatting a file, and it needs accurate SCM / git-blame data.

  Beneath both halves, a strip titled "Suppression is a claim that needs evidence — every lever, per tool" gives one cell each for SpotBugs 4.10.2 (@SuppressFBWarnings(value="EI_EXPOSE_REP", justification="…") from edu.umd.cs.findbugs.annotations, CLASS retention, bytecode), PMD 7.25.0 (// NOPMD with trailing justification text; @SuppressWarnings("PMD.RuleName")), Checkstyle 13.6.0 (requires SuppressWarningsFilter + SuppressWarningsHolder under TreeWalker, then @SuppressWarnings("checkname")), Error Prone / NullAway (@SuppressWarnings("CheckName"); castToNonNull(x); -Xep:Check:OFF|WARN|ERROR, last flag wins), and SonarQube Server 2026.1 LTA (//NOSONAR rule-blind; reviewed False Positive or Accepted transition, where "Accepted" supersedes "Won't Fix"). A legend maps the four ladder shades from Lever 1 (narrowest, one site) to Lever 4 (future code, ratchet).
