Fig 39.1 — Finding triage and the four-lever scope ladder

Every finding demands one triage decision; each decision maps to exactly one lever ordered narrow→broad. Breadth is always a smell.

Breadth is always a smell. Reaching for a broad lever (class-wide @SuppressWarnings, rule-blind //NOSONAR) when only one site or one rule is wrong silences future real findings in that scope. The discipline: always reach for the narrowest lever that solves the actual problem.
Step 2 — apply the right lever (narrow → broad)
LEVER 1 Per-finding suppression scope: one site
SpotBugs: @SuppressFBWarnings(value="NP_…", justification="…") — justification field is intentional.
PMD: @SuppressWarnings("PMD.RuleName") or // NOPMD <reason> (trailing text enters report).
Checkstyle: @SuppressWarnings("checkname") (requires SuppressWarningsFilter + SuppressWarningsHolder on TreeWalker).
Error Prone: @SuppressWarnings("CheckName").  NullAway: castToNonNull(x) at the exact site.
Sonar: //NOSONAR (EOL; rule-blind — drops all issues on the line); prefer reviewed False Positive / Accepted transition.
use when
The tool is wrong at exactly this one site; reason is traceable.
danger
Broad form (@SuppressWarnings("PMD") on a class) silences all future findings of that kind in scope.
↓
if the same rule misfires everywhere in the project →
LEVER 2 Rule / ruleset tuning scope: one rule, project-wide
Checkstyle: drop the module or set severity.
PMD: <exclude name="RuleName"/>; or violationSuppressRegex / violationSuppressXPath for systematic misfires by message or AST shape.
SpotBugs: FindBugsFilter Match with no class/method narrowing = "never report this pattern."
Error Prone: -Xep:CheckName:OFF (last flag for a check wins).
Sonar: deactivate the rule in the Quality Profile.
use when
One rule misfires the same way across the whole project; one config change replaces scattered site suppressions.
danger
Global OFF removes signal where the rule would have caught a real defect. Do not use to make a gate green fast under deadline.
↓
if legacy debt is too large to fix now →
LEVER 3 Baseline — freeze existing findings scope: all current findings
SpotBugs: baselineFiles — "bugs found in the baseline files won't be reported" (multi-file since spotbugs-maven-plugin 4.7.1.0).
Checkstyle: external suppressions.xml via SuppressionFilter (one <suppress> per frozen finding = de-facto baseline); maxAllowedViolations caps by count.
Sonar: anything outside the New Code window is tracked but excluded from the default gate.
use when
Legacy codebase is too noisy to fix on day 1; gate the delta not the total. Precondition for turning a gate on at all.
danger
Freezes real bugs hiding among pre-existing findings. Count cap is order-blind; finding-set baseline drifts on refactor.
↓
combine with lever 3: let past persist, block new →
LEVER 4 Ratchet — "clean as you code" scope: future code only
Sonar: New Code Definition marks what is "new"; the default gate applies its conditions only to new code. Legacy does not block the build; new code must be clean; debt decays as files are touched.
Checkstyle: maxAllowedViolations set to today's count, then lowered over time = count ratchet; blocks any increase.
SpotBugs: regenerating the baseline at each release plays the same role at the finding-set level.
use when
Baseline (lever 3) is in place. Combine: lever 3 freezes past; lever 4 blocks new debt. Enables adoption on a million-line legacy codebase with no flag-day.
danger
Never forces legacy remediation; cold modules stay dirty indefinitely. Count ratchets can be gamed (split a file, reformat). Needs accurate SCM / git-blame data.
Suppression is a claim that needs evidence — every lever, per tool
