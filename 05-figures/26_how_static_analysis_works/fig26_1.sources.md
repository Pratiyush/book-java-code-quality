- Long-description: A diagram titled "Fig 15.1 — Static-analysis technique ladder," subtitled that there are four moves, each seeing more than the one below, at rising power and cost. The left side is a stacked ladder of four rung cards read top to bottom, each joined to the next by a downward arrow that names what the next move adds. Each rung carries a depth bar (lightening to darkening with the move number), a move number and name, a cost badge, a "runs on/in" pill, example tools, and two columns: "catches well" and "blind spot." Move 1, "Parse to an AST — match shapes," cost low, runs on source; tools PMD (XPath / visitor on the root AST node), Checkstyle (source tokens), Error Prone (compiler AST inside javac); catches well style, naming, syntactic smells, empty catch, missing break; blind spot anything behavioral — it sees shape, not runtime values. The connector to Move 2 reads "+ symbols & types." Move 2, "Resolve symbols & types," cost low, runs on source / compiler; tools PMD SymbolFacade plus optional TypeResolution, Error Prone (full types free inside javac); catches well type-aware misuse such as a wrong overload, raw types, LinkedList.get(int) in a loop; blind spot flow-dependent facts — it cannot tell if a value is null at a point. The connector to Move 3 reads "+ control-flow graph + data-flow fixpoint." Move 3, "Control- & data-flow analysis," cost medium-high, runs on source / bytecode / whole-program graph; tools PMD DFA visitor building control flow graphs and data flow nodes, SpotBugs OpcodeStackDetector on bytecode, CodeQL local/global data-flow graph; catches well null-deref, unclosed resource, lock discipline intraprocedurally, and leaks/injection spanning methods inter- or globally; blind spot intraprocedural analysis stops at the method boundary, and global flow must approximate aliasing, reflection, and dynamic dispatch. The connector to Move 4 reads "+ source / sink / sanitizer roles." Move 4, "Taint tracking" (drawn with a heavier border as the top rung of power), cost high, runs in CI / nightly, not pre-commit; tools CodeQL — taint extends data flow to include steps where data values are not necessarily preserved but the potentially insecure object is still propagated (for example y = x+1, y derived from x) — and Semgrep — AST to IL plus constant-propagation plus taint, with no path sensitivity, no pointer or shape analysis, and no soundness guarantees; catches well SQLi, XSS, and command injection from an attacker-controlled source to a dangerous sink; blind spot unmodeled sources or sanitizers, and the Semgrep engine being intraprocedural. The right side holds two blocks. A "power & cost" block shows a four-cell ramp from light at the bottom to dark at the top, labelled "↑ more power" above and "↓ cheaper" below, with the note that Move 1 runs on every keystroke while Move 4 runs nightly in CI. A "Soundness ↔ Completeness" block defines Sound (a filled dot: no false negatives, all real bugs reported, at the cost of false positives) and Complete (a hollow dot: no false positives, every report real, at the cost of false negatives), notes that no analyzer is both for a non-trivial property (Rice's theorem leading to the halting problem), and quotes Checker Framework ("values soundness over limiting false positives") and Semgrep OSS ("No soundness guarantees").
