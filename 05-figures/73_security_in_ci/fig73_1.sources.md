- Long-description: The figure is a horizontal four-stage pipeline read left to right, with arrow connectors between stages. Stage 1 (Pre-commit): one check row labeled "Secrets" — scans code, diffs, and history for leaked credentials; the only stage that can prevent a leak. Policy strip: BLOCK — any detected secret prevents the credential reaching the repo. Stage 2 (Pull Request / Build, highlighted with an orange left border as the shift-left stage): three check rows separated by horizontal dividers. Row 1 labeled "Secrets" — CI re-scan of pushed diff and repo history (push-protection layer). Row 2 labeled "SAST" — static analysis of application code for injection, crypto misuse, unsafe deserialization, naming tools Semgrep 1.163.0, SpotBugs 4.10.2 + FindSecBugs, CodeQL. Row 3 labeled "SCA" — dependency composition scan for known CVEs and license issues, naming tools OWASP Dependency-Check 12.2.2, Grype 0.108–0.110, Trivy 0.71.0. Policy strip: BLOCK ON HIGH-SEVERITY NEW FINDINGS — PR not blocked by pre-existing low-severity debt it did not introduce; route findings to security reviewer for exploitability judgment. Stage 3 (Continuous monitoring, async): one check row labeled "SCA" — scheduled re-scan for newly disclosed CVEs in unchanged dependencies using Renovate / Dependabot and SBOM CycloneDX 1.6. Policy strip (dashed border): WARN → triage — new CVE disclosures routed to backlog or security reviewer; does not block the PR pipeline. Stage 4 (Staging / deployed app, release gate): two check rows separated by a divider. Row 1 labeled "DAST" — black-box probe of the running app for runtime misconfiguration and environment-specific flaws SAST cannot reach, citing OWASP ZAP. Row 2 labeled "IAST" — instrumented app under test; runtime context reduces false positives; a SAST/DAST hybrid; needs instrumentation agent. Policy strip: BLOCK RELEASE ON HIGH-SEVERITY FINDINGS — slow, needs deployed app and test scenarios; small internal apps may rationally omit. Below the pipeline: a legend with three swatches — orange-accented for "Shift-left stage, fast, runs every PR"; solid-bordered for "BLOCK — fails the build / merge / release"; dashed-bordered for "WARN → triage — routed; does not block." A limit note (orange left border) reads: "Green gate ≠ secure. A clean scan means no detected known issue — tools miss logic and authorization flaws, CVE databases miss zero-days, and a scan is not a penetration test. Gate fatigue is the primary failure mode: blocking on every finding produces a permanently-red build that teams disable. Block narrowly (high-severity, new code); warn and triage everything else."
