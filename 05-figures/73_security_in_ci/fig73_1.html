Figure 32.1 — The security gate: five testing types ordered fast-to-slow

Static checks (secrets, SAST, SCA) run at pre-commit and PR; dynamic checks (DAST, IAST) gate the release against staging. Block only high-severity new findings; warn and triage the rest.

Stage 1
Pre-commit
Secrets Scans code, diffs, and history for leaked credentials — the only stage that can prevent a leak
BLOCK
Any detected secret — prevents the credential ever reaching the repo
→
Stage 2 · shift-left
Pull Request / Build
Secrets CI re-scan of pushed diff and repo history (push-protection layer)
SAST Static analysis of application code — injection, crypto misuse, unsafe deserialization (e.g. Semgrep 1.163.0, SpotBugs 4.10.2 + FindSecBugs, CodeQL)
SCA Dependency composition scan — known CVEs in libraries + license issues (e.g. OWASP Dependency-Check 12.2.2, Grype 0.108–0.110, Trivy 0.71.0)
BLOCK on high-severity NEW findings
Clean-as-you-code scope: PR is not blocked by pre-existing low-severity debt it did not introduce. Route finding to security reviewer for exploitability judgment.
→
Stage 3 · async
Continuous monitoring
SCA Scheduled re-scan for newly disclosed CVEs in unchanged dependencies (Renovate / Dependabot + SBOM CycloneDX 1.6)
WARN → triage
New CVE disclosures routed to backlog or security reviewer; does not block the PR pipeline
→
Stage 4 · release gate
Staging (deployed app)
DAST Black-box probe of the running app — runtime misconfiguration and environment-specific flaws SAST cannot reach (e.g. OWASP ZAP)
IAST Instrumented app under test — runtime context reduces false positives vs pure static; a SAST/DAST hybrid. Needs instrumentation agent.
BLOCK release on high-severity findings
Slow; needs deployed app + test scenarios. Small internal apps may rationally omit — cost and setup are real.
Shift-left stage — fast, runs every PR
BLOCK — fails the build / merge / release
WARN → triage — routed; does not block
Green gate ≠ secure. A clean scan means "no detected known issue" — tools miss logic and authorization flaws (need threat modeling + design review), CVE databases miss zero-days, and a scan is not a penetration test. Gate fatigue is the primary failure mode: blocking on every finding produces a permanently-red build that teams disable. Block narrowly (high-severity, new code); warn and triage everything else.
