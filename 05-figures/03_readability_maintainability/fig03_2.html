Figure 2.2 — Contested prescriptions: function size & comments

Two reputable schools hold opposed positions. Neither is universally correct — choose deliberately, apply consistently.

School A

Clean Code

Robert C. Martin (2008)

Function size
Position: very small functions. "Functions should be small… smaller than that." Each function does one thing only. Extraction is always preferred when a piece of logic could be named.
Strongest limitation: fragmentation — following one idea requires jumping across many small methods; adds indirection and naming overhead.
Comments
Position: comments are largely failures. "A comment is an apology for not making the code self-explanatory." Good naming and structure make most comments redundant; comments that explain what code does are a smell.
Strongest limitation: comments that capture why (design intent, rejected alternatives) cannot always be expressed in code alone; without them, rationale is lost.
Shared commitment
Strong naming, zero side-effects per function, consistency within a codebase.

School B

A Philosophy of Software Design

John Ousterhout (2018; 2nd ed. 2021)

Function size
Position: deep modules preferred. A module with a simple interface over a substantial implementation is "deep" — it hides complexity without forcing callers to understand it. Excessive decomposition makes the interface cost as high as the implementation.
Strongest limitation: deep modules can concentrate complexity in ways that make individual functions difficult to test or understand in isolation.
Comments
Position: comments are essential. Comments capture design intent and the why behind decisions — information the code itself cannot express. Interface comments ("what does this do") and implementation comments ("why this approach") both have value.
Strongest limitation: comments diverge from code over time; a misleading stale comment is worse than no comment.
Shared commitment
Local reasoning, reducing cognitive load, consistent application of whichever style the team adopts.
 
Both schools agree on the agreed core (low-controversy)
Naming carries most of the load Names are the densest documentation in the code. Neither school disputes this.
Local reasoning lowers cognitive load Code you can understand without holding the whole system in mind is more readable.
Consistency reduces load Whichever style is chosen, apply it consistently — that is why formatters and linters exist.
How to use this map: treat the disagreements as context-dependent trade-offs, not winners. Tiny functions aid navigation but can fragment a readable algorithm; comments rot but capture why; var cuts noise but can hide types. A team picks a position deliberately and applies it consistently (Chapters 6, 37). The contested zone does not have a correct answer; it has a correct process.
