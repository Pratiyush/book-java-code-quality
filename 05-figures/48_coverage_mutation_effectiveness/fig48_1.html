Fig 48.1 — Coverage vs mutation: the assertion gap on the same covered line

100% line coverage does not move when assertions are added — mutation score does. Coverage records execution; mutation score records detection.

Weak test — assertion-free BEFORE
// asserts only non-null — no boundary, no value @Test void discount_returnsValue() { var result = Discount.apply( 10, 100.0, 0.10); assertNotNull(result); // ← only assertion }
JaCoCo LINE 100 % PASS
JaCoCo BRANCH < 100 % FAIL gate
Mutation score low FAIL gate
Mutator (PITest DEFAULTS) Change introduced Status
CONDITIONALS_BOUNDARY >= → > on qty >= 10 SURVIVED
MATH * → / in price * (1 - rate) SURVIVED
VOID_METHOD_CALLS delete void call (early return) SURVIVED
NULL_RETURNS return null instead of result SURVIVED
What went wrong: the test executed every line (LINE = 100%) but asserted only that the return value is non-null. A flipped boundary, a swapped operator, or a deleted call all return a non-null value — each mutant survived. PITest's own statement: coverage "does not check that your tests are actually able to detect faults in the executed code."
Strong test — boundary + value assertions AFTER
// AssertJ: boundary and computed value both checked @Test void discount_appliesAboveBoundary() { assertThat(Discount.apply(10, 100.0, 0.10)) .isCloseTo(90.0, within(0.001)); // value assertThat(Discount.apply(9, 100.0, 0.10)) .isCloseTo(100.0, within(0.001)); // boundary }
JaCoCo LINE 100 % PASS
JaCoCo BRANCH 100 % PASS gate
Mutation score high PASS gate
Mutator (PITest DEFAULTS) Change introduced Status
CONDITIONALS_BOUNDARY >= → > on qty >= 10 KILLED
MATH * → / in price * (1 - rate) KILLED
VOID_METHOD_CALLS delete void call (early return) KILLED
NULL_RETURNS return null instead of result KILLED
What changed: LINE coverage stayed at 100% — the same lines ran. BRANCH coverage rose to 100% (both sides of qty >= 10 now exercised). Mutants were killed because the AssertJ value assertions failed on each mutated return. Coverage did not move; assertion strength did.
The two-tool answer to two different questions
JaCoCo org.jacoco:jacoco-maven-plugin Question: what did the tests touch?
Gate every commit — cheap (single instrumented run, Java agent). Read BRANCH (C1), not only LINE: a line counts covered when one instruction ran; BRANCH exposes the untested else-path LINE hides. Check goal: element BUNDLE, counter BRANCH, value COVEREDRATIO, minimum <threshold>, haltOnFailure default true. JaCoCo 0.8.16 (Java 21 support: 0.8.11; Java 25: 0.8.14).
PITest org.pitest:pitest-maven goal mutationCoverage Question: would the tests catch a bug?
Run periodically — expensive (covering tests once per mutant). Statuses: KILLED / SURVIVED / NO_COVERAGE / TIMED_OUT / MEMORY_ERROR / RUN_ERROR / NON_VIABLE. Mutation score = killed ÷ total. Test strength = killed ÷ mutants-with-coverage. JUnit 5 (Jupiter) requires pitest-junit5-plugin — absent = silent no-coverage. PITest 1.25.3.
The invariant: LINE coverage did not move between the two tests above — the executed code was identical. Mutation score did, because what changed was whether the tests checked the result. High coverage with weak assertions manufactures false confidence. Neither metric is "test quality" alone: coverage is a gap-finder (necessary floor); mutation score is a fault-detection gauge (stronger signal, higher cost). Run both; crown neither.
