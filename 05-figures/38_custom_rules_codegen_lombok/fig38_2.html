Fig 38.2 — Codegen approaches by relation to the JSR 269 standard contract

Three positions on one spectrum: compiler-derived → new-file processor (inside contract) → AST-edit processor (past contract). Each states its trade-off; none is crowned.

Position A — Sidesteps the contract
Language-level derivation
  • record (JEP 395, final JDK 16)
  • Compiler derives canonical ctor, accessors,
    equals, hashCode, toString
  • No dependency  ·  No processor  ·  No generated file
Position B — Inside the contract
Generate-new-files processors
  • AutoValue @AutoValue → AutoValue_Foo
  • Immutables @Value.Immutable → ImmutableFoo
  • MapStruct @Mapper → FooMapperImpl
  • Standard Filer API  ·  Visible, inspectable output file

AST-edit processor
  • Lombok — edits annotated type in place
  • Casts to com.sun.tools.javac.* internal types
  • Forces extra rounds via dummy file + Filer patch
  • No documented mutate-existing-type method in JSR 269

Neutral framing (NEUTRALITY.md): record, generate-new-files processors, and Lombok are different approaches to the same boilerplate problem — a codebase may use several at once (records for DTOs, MapStruct for mapping, Lombok for logging). Each states its trade-off; none is crowned. The live question is which mix matches a team’s surface area and its tolerance for the respective coupling.
Annotation processing note
annotationProcessorPaths in maven-compiler-plugin becomes mandatory at JDK 23; before that, a processor on the plain classpath was discovered implicitly. Multi-processor builds need explicit ordering (lombok-mapstruct-binding). Routing: JaCoCo @Generated exclusion → metrics chapter. record as a full language feature → Ch 5 / Ch 8.
