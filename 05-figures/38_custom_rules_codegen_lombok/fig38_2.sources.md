## Accessibility
- Alt-text: Concept-map showing three codegen positions on a spectrum — record (sidesteps JSR 269, compiler derives members), AutoValue/Immutables/MapStruct (inside the contract, generate a new visible file), and Lombok (past the contract, edits the annotated type via javac internals) — each with its strongest case and hardest limitation, and none crowned.
- Long-description: The figure opens with a three-zone spectrum bar. The left zone, labelled Position A — Sidesteps the contract, shows record (JEP 395, JDK 16): compiler derives canonical constructor, accessors, equals, hashCode, and toString; no dependency, no processor, no generated file. The centre zone, labelled Position B — Inside the contract, lists AutoValue (@AutoValue → AutoValue_Foo), Immutables (@Value.Immutable → ImmutableFoo), and MapStruct (@Mapper → FooMapperImpl), all using the standard Filer API and producing a visible output file. The right zone, labelled Position C — Past the contract, describes Lombok: it edits the annotated type in place by casting to com.sun.tools.javac.* internal types and forces extra compiler rounds via a dummy file and Filer patch; there is no documented mutate-existing-type method in JSR 269. Below the spectrum, a four-cell strip summarises the JSR 269 round model: SPI discovery via META-INF/services, the round loop calling process(annotations, RoundEnvironment) ("query about a round of annotation processing"), Filer creation of new files ("creation of new files by an annotation processor", SE 21 verbatim), and the contract limit of no documented method to modify an already-declared type. Below that, three detail cards repeat the three positions. The record card states strongest case (no dependency, no processor, transparent carrier with compiler-derived value semantics) and hardest limit (transparent carrier only: cannot extend a class, shallowly immutable; builders, mutable entities, mappers, loggers out of scope). The AutoValue/Immutables/MapStruct card states strongest case (generated class is ordinary inspectable Java; covers value types with builders, mappers, optional fields) and hardest limit (caller references a generated symbol; generated-sources dir needed; processor ordering unspecified; lombok-mapstruct-binding for multi-processor builds). The Lombok card (with heavier border, indicating most contested — not winner) states strongest case (maximal terseness: @Data, @Builder, @Slf4j, @SneakyThrows, @Cleanup; addLombokGeneratedAnnotation=true → JaCoCo ≥ 0.8.1 excludes @lombok.Generated members) and hardest limit (depends on com.sun.tools.javac.* internals; --add-opens required since JDK 16; source the developer reads ≠ code that runs; IDE plugin required; annotationProcessorPaths mandatory at JDK 23; delombok is the exit). At the bottom, a callout note states the neutral framing: record, generate-new-files processors, and Lombok are different approaches to the same boilerplate problem; a codebase may use several at once; none is crowned. A smaller annotation-processing note box records that annotationProcessorPaths becomes mandatory at JDK 23, with routing notes to the metrics chapter (JaCoCo @Generated exclusion) and Ch 5/Ch 8 (record as language feature).
