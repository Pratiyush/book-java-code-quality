## Accessibility

- Alt-text: Pact's four-stage pipeline flowing left to right: consumer test, publish to broker, provider verification, then a can-i-deploy gate.

- Long-description: The figure, titled in its caption "Pact four-stage pipeline" (referenced in the prose as Figure 24.1) and subtitled that the consumer-driven contract guarantee requires all four stages because any half-pipeline delivers false confidence, is a left-to-right flow of four stage boxes joined by arrows, with an artifact label under each.

  Stage 1, Consumer test: uses @PactConsumerTestExt, a @Pact method declares the interaction, PactDslJsonBody does type-matching (stringType(), matchRegex()), the real client calls an injected MockServer, and it writes the pact JSON to target/pacts; it runs against the mock, not the real provider. Its artifact is the pact JSON in target/pacts. An arrow leads to Stage 2, Publish to broker, drawn with a dashed border to mark it as shared infrastructure: it is the shared store of contracts plus verification results, holds consumer version selectors (deployed, released), and builds the consumer Г— provider matrix; without a broker there is no can-i-deploy. Its artifact is "broker matrix updated." An arrow leads to Stage 3, Provider verification: uses @Provider and @PactBroker, @TestTemplate generates one test per interaction, verifyInteraction() replays each request against the running provider, @State seeds preconditions, and it checks the real response matches the contract; it holds the consumer's mock honest and needs a live provider. Its artifact is "verification result в†’ broker." An arrow leads to Stage 4, the can-i-deploy gate: it queries the broker matrix of consumer Г— provider verification results, uses the deployedOrReleased selector, calls record-deployment / record-release after a successful deploy, and treats deployment as safe only when all consuming versions are verified against this provider version; the gate is only meaningful with record-deployment discipline. Its artifact is "deploy / block."

  A callout below names the half-pipeline failure mode: Stage 1 alone proves only the consumer's assumptions вЂ” the mock can expect data the real provider never sends вЂ” and Stage 3 is what holds the mock honest, so all four stages must run or none of the contract guarantee applies. A legend distinguishes a solid border (a participant stage: consumer or provider) from a dashed border (shared infrastructure: the Pact Broker), and gives the GAV coordinates au.com.dius.pact.consumer:junit5 4.4.x and .provider:junit5 4.6.x (versioned independently).
