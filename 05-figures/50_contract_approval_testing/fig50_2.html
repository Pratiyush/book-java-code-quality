Figure 24.2 — Three techniques, three questions on the service boundary and its output

Each answers what the others cannot; each fails when its own reference goes wrong.

Service boundary — two jobs, not rivals
Complex output
TECHNIQUE 1 · SERVICE BOUNDARY
Contract testing
Tool: Pact (pact-jvm)
Question answered Do consumer and provider still agree on message shape?
Needs live partner? No — each side in isolation. Consumer runs against MockServer; provider replays recorded pact.
Produces artifact? Yes — pact JSON + broker matrix of verification results.
Pyramid band middle
Scope limit (verbatim, Pact docs) "Not suitable for public or third-party APIs." Not a functional, performance, or load test.
Central risk Consumer pact without provider verification is the over-mocking anti-pattern: the consumer's own assumptions with a broker behind them. Run all four stages or none of the guarantee applies.
TECHNIQUE 2 · SERVICE BOUNDARY
API / HTTP testing
Tool: REST-assured 5.5.0
Question answered Does the running endpoint respond correctly?
Needs live partner? Yes — a running endpoint (Testcontainers / framework test instance).
Produces artifact? No artifact — asserts on the live HTTP response.
Pyramid band middle → top
DSL given() / when() / then(); body path = GPath (Groovy GPath, not Jayway JsonPath).
Central risk Cannot see a downstream consumer break — no consumer is involved. Cannot replace a contract test; cannot replace a unit test of business logic.
TECHNIQUE 3 · OUTPUT
Approval testing
Tool: ApprovalTests.Java
Question answered Did the output change from a human-reviewed baseline?
Mechanism Approvals.verify(result) writes *.received.*, compares *.approved.*; mismatch launches diff/reporter. Scrubbers normalize non-determinism (timestamps, GUIDs).
Fits best for Large / hard-to-hand-write output; golden-master for legacy characterization before refactor.
Pyramid band any level
Approved files Committed to version control — surface in pull-request review as second-reviewer check.
Central risk Verifies "unchanged," not "correct." Rubber-stamping a wrong received file bakes the bug into the baseline. Without diff discipline, it is theatre.
Unifying caveat: every reference-based test is only as good as its reference. A wrong contract (unverified consumer mock), a blind API assertion (no downstream consumer view), or a rubber-stamped baseline each turns the test into theatre. These three compose — REST-assured proves endpoints run, Pact proves no consumer is broken, ApprovalTests pins complex outputs — and none replaces unit/property tests that assert business-logic invariants.
