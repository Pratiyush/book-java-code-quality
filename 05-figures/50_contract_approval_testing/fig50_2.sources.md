**One idea:** Three complementary techniques each answer a question the others cannot — contract testing asks "do the sides agree?" (no live partner), API testing asks "does the endpoint behave?" ## Accessibility

- Alt-text: Three side-by-side technique cards — contract testing, API/HTTP testing, and approval testing — each with the question it answers and its central risk.

- Long-description: The figure, titled in its caption "Three techniques, three questions on the service boundary and its output" (referenced in the prose as Figure 24.2) and subtitled that each answers what the others cannot and each fails when its own reference goes wrong, places three technique cards in a row. A bracket above the first two cards labels them "Service boundary — two jobs, not rivals"; a separate dashed bracket above the third labels it "Complex output."

  Card 1, Technique 1 (service boundary), Contract testing, tool Pact (pact-jvm): question answered — do consumer and provider still agree on message shape? Needs a live partner? No — each side in isolation, the consumer runs against MockServer and the provider replays the recorded pact. Produces an artifact? Yes — the pact JSON plus the broker matrix of verification results. Pyramid band: middle. Scope limit, verbatim from Pact docs: "Not suitable for public or third-party APIs," and not a functional, performance, or load test. Central risk: a consumer pact without provider verification is the over-mocking anti-pattern — the consumer's own assumptions with a broker behind them — so run all four stages or none of the guarantee applies.

  Card 2, Technique 2 (service boundary), API / HTTP testing, tool REST-assured 5.5.0: question answered — does the running endpoint respond correctly? Needs a live partner? Yes — a running endpoint (Testcontainers or a framework test instance). Produces an artifact? No — it asserts on the live HTTP response. Pyramid band: middle to top. DSL: given() / when() / then(), with the body path expressed in GPath (Groovy GPath, not Jayway JsonPath). Central risk: it cannot see a downstream consumer break because no consumer is involved, and it can replace neither a contract test nor a unit test of business logic.

  Card 3, Technique 3 (output), Approval testing, tool ApprovalTests.Java, drawn with a dashed border to mark it as the output technique: question answered — did the output change from a human-reviewed baseline? Mechanism: Approvals.verify(result) writes a *.received.* file and compares it to the *.approved.* file; a mismatch launches a diff/reporter, and scrubbers normalize non-determinism such as timestamps and GUIDs. Fits best for large or hard-to-hand-write output, and as a golden-master for legacy characterization before a refactor. Pyramid band: any level. Approved files: committed to version control and surfaced in pull-request review as a second-reviewer check. Central risk: it verifies "unchanged," not "correct" — rubber-stamping a wrong received file bakes the bug into the baseline, and without diff discipline it is theatre.

  A bottom note states the unifying caveat: every reference-based test is only as good as its reference, so a wrong contract (an unverified consumer mock), a blind API assertion (no downstream consumer view), or a rubber-stamped baseline each turns the test into theatre; the three compose — REST-assured proves endpoints run, Pact proves no consumer is broken, ApprovalTests pins complex outputs — and none replaces the unit and property tests that assert business-logic invariants. A legend distinguishes a solid border (a boundary technique, needs a service) from a dashed border (an output technique, any unit or integration scope), and gives the GAV coordinates io.rest-assured:rest-assured 5.5.0 (2024-07-05), au.com.dius.pact.consumer:junit5 4.4.x, and github.com/approvals/ApprovalTests.Java.
