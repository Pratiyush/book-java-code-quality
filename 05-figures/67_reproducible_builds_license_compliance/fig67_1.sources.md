## Accessibility
- Alt-text: A three-column chain showing five Java build non-determinism sources, the specific fix for each, and a four-step verify-by-rebuild panel, with a hermeticity note and an honest-limits strip.
- Long-description: The figure flows left to right in three columns. The left column lists five sources of non-determinism, top to bottom: JAR entry and MANIFEST timestamps; archive file ordering; unpinned dependency or plugin versions; locale, timezone, or encoding; and embedded build-time data. An arrow column maps each to a fix in the middle column: a fixed `project.build.outputTimestamp` with the reproducible-build-maven-plugin; Gradle `reproducibleFileOrder = true` and `preserveFileTimestamps = false`; pinning exact dependency and plugin versions plus the wrapper; fixing LANG, TZ, and file.encoding in the CI environment; and avoiding embedded build-time data. A final arrow leads to the right column, a "Verify by rebuilding" panel with four numbered steps: build twice on a different machine and time; diff byte-for-byte (Maven `artifact:compare`); publish checksums and signatures; and run rebuild-and-diff as a CI step. Below the verify panel a note defines the strong form, hermeticity (only declared pinned inputs, no network fetch, required for higher SLSA levels). A bottom strip lists four honest limits: last-mile effort, toolchain variance, integrity-not-correctness, and silent decay. The caption states the content is factual only and not legal advice.
