Figure 29.1 — The reproducibility chain

Five sources of Java build non-determinism, their fixes, and the verify step that closes the chain.

Source of non-determinism
The fix
Verify by rebuild
JAR entry & MANIFEST timestamps Current build time embedded in every archive entry; differs each run
Archive file ordering Non-deterministic filesystem enumeration order inside the ZIP/JAR
Unpinned dep / plugin versions Version ranges or SNAPSHOT deps resolve to different releases on different machines (Ch 27)
Locale / timezone / encoding LANG, TZ, file.encoding differ between CI and developer machine
Embedded build-time data Generated code or resources that include build dates, absolute paths, or random values
→ → → → →
Maven: <project.build.outputTimestamp> Set to a fixed ISO-8601 date; reproducible-build-maven-plugin normalises JAR entry metadata
Gradle: reproducibleFileOrder = true Also preserveFileTimestamps = false in the archive task block
Pin exact versions — deps AND plugins No ranges ([1.0,)), no SNAPSHOT; lock the build tool via the wrapper (Ch 27)
Fix LANG / TZ / file.encoding in CI env Declare them explicitly in the CI job environment; match local toolchain setup
Avoid embedding build-time / env data Do not write current date, hostname, or random seed into generated artefacts or resources
→
Verify by rebuilding
1
Build the artifact twice — different machine, different time, same source tag
2
diff the two outputs byte-for-byte;
Maven: artifact:compare goal automates the comparison
3
Publish checksums + signatures so any third party can repeat the rebuild (link to Ch 28 provenance)
4
Run rebuild-and-diff as a CI step — treat a byte difference as a build failure; reproducibility decays silently otherwise
Strong form — Hermeticity
Build depends only on declared, pinned inputs: no network fetch of moving deps at build time. Required for higher SLSA levels (Ch 28).
Honest limits
Last-mile effort. Chasing truly bit-identical output can require hunting obscure non-determinism; diminishing returns for a purely internal app.
Toolchain variance. Different JDK builds/vendors can still differ; full reproducibility may require pinning the JDK itself.
Integrity, not correctness. A reproducible build of buggy code is reliably, verifiably buggy. Proves determinism and anti-tampering, not quality.
Decays silently. One unpinned plugin or new timestamp-embedding tool breaks reproducibility without any test failing — the CI verify step is mandatory.
