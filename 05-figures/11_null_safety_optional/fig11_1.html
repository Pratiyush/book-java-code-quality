Fig 11.1 — Null-safety: four levers of layered defense

Each lever catches the null the earlier one cannot reach. Detection moves left (earlier) across the lifecycle.

Earlier
Detection point
Later
1
Optional — absence as a type Optional.of(v) / .ofNullable(v)
design-time
What it does

Models "no result" in the return type; the caller cannot ignore the absent case. Optional.of(v) throws if null; .ofNullable(v) yields empty. Effective Java Items 54–55: return empty collections / Optional, never null.

java:S2789 OptionalNotPresent java:S3655
Hardest limit

Allocates; unsuited to hot paths (Item 55). Anti-pattern: Optional fields, parameters, collection elements, Optional<List<T>>, .get() without a guard. Never nest Optional in a container or use as a map key.

↓ null that slips past design passes to the boundary
2
requireNonNull — fail fast at the boundary Objects.requireNonNull(obj, "msg")
boundary (runtime)
What it does

Converts a silent later NPE into a loud, immediate failure at the entry point, naming the offending argument. requireNonNullElse(obj, default) returns a non-null fallback. java.util.Objects — in Java since Java 7 / Java 9.

Hardest limit

Still throws; shifts failure earlier, not absent. Invisible to the compiler (no type guarantee). Do not reflexively guard every private/internal method where the invariant already holds.

↓ null that passes the boundary reaches the build — annotations + checker catch it here
3
Annotations + checker — "might NPE" becomes a build error
build-time
What it does

Annotation vocabulary (JSpecify 1.0.0: @Nullable / @NonNull / @NullMarked / @NullUnmarked) + a checker (NullAway: fast, deliberately unsound, ~1.15× build cost; Checker Framework Nullness Checker: sound, ~5.1× build cost). @NullMarked on package-info.java = everything non-null by default.

-Xep:NullAway:ERROR @NullMarked ReturnMissingNullable
Hardest limit

Annotations alone are inert without a wired checker. No checker catches reflection, deserialization, or JNI nulls. NullAway is unsound (deliberate false negatives). Checker Framework soundness carries annotation effort + build cost. Field-init strictness surprises DI code.

↓ null that escapes all static reasoning reaches the JVM — JEP 358 makes it diagnosable
4
JEP 358 — helpful NPE messages on by default since JDK 15
runtime
What it does

JVM analyzes the bytecode at the throw site and names the exact null expression ("Cannot invoke "String.toLowerCase()" because the return value of "Customer.getEmail()" is null"). Active at Java 21/25 with no flag. Makes the surviving NPE diagnosable in one read.

Hardest limit

Diagnoses; prevents nothing. Names exposed in the message may appear in logs (minor information-disclosure consideration). The latest line of defense — the honest counterpoint to "every null can be designed out."
