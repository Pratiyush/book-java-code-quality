Figure 39.1 — The safe-change loop: one invariant at four scales

Preserve behavior · verify with tests · move in small reversible steps · keep the system working throughout

0

Precondition — ensure a test net exists

Tests cover the area. On untested legacy: build the net first (characterization tests + seams). A characterization test pins current behavior — including bugs — as a net, not a specification; review what is captured.

Feathers WELC 2004 — "legacy code is code without tests"
1

Make ONE small behavior-preserving transformation

Apply a single refactoring (Extract Method · Rename · Move · Replace Conditional with Polymorphism · Introduce Parameter Object). One step only. Prefer IDE-automated refactorings — behavior-preserving by construction.

Fowler Refactoring 2e (2018) — catalog entries
✗

Never: refactor + behavior change in one step

Fowler "two hats" — at any moment: refactor (structure fixed, behavior preserved) OR add feature (behavior changes, structure fixed). Mixing them means a failing test cannot localize which change broke it.

2

Run the test suite — must stay green

The suite is the verification gate. A failure here means behavior changed: revert the step, do not proceed. Green = behavior preserved.

Fowler Refactoring 2e — "suite runs after every refactoring"
3

Commit — one commit per step

Each commit is independently reversible. The codebase is never more than one small step away from a known-good state. This is the safety that the big-bang rewrite discards.

↩ Repeat — return to step 1
Preparatory refactoring — refactor before the feature to make the change easy (Fowler), and clean up as code is touched (Boy Scout Rule, Ch 1). Never "while I'm here" — refactoring belongs in its own PR, under its own hat.

The same loop at four scales

A
Method / class — Refactoring Small behavior-preserving transforms under a green suite. IDE refactorings are behavior-preserving by construction. Some 2018 catalog entries are now language features: records, sealed + pattern matching (Ch 5). Limit: refactoring without tests is just editing.
B
Untested legacy — Characterize & seam Break deps to a seam (object / interface / link) using only IDE behavior-preserving refactorings (Extract Interface · Parameterize Constructor · Subclass-and-Override). Write a characterization test. Then apply scale A. Loop: find change point → seam → characterize → change → refactor green. Limit: characterization pins current behavior including bugs — review the captured output; it is a net, not a spec. (Feathers WELC 2004)
C
Whole system — Strangler fig Façade / routing layer in front of old system → build new slices behind it → redirect calls old→new per feature/endpoint → retire old when nothing routes to it. Feature flags toggle each slice. Contract tests + characterization at the seam. (Fowler 2004) Limit: half-strangled stall is the failure mode — dual-running cost + shared data/state is the hard part; needs org stamina to finish.
D
Platform — Java version migration LTS-to-LTS path: 8 → 11 → 17 → 21 → 25. Process: tests green on current → dependencies first (usual blocker: old lib/agent) → run UpgradeToJava21 recipe (OpenRewrite 8.81.0) → fix residual breaks (JPMS, removed APIs) → build on target JDK matrix → modernize after (records / sealed / pattern matching) as a separate step. Limit: migration ≠ modernization — bumping the version does not refactor old idioms; recipes do not cover everything; preview at 25 is not a migration target.

The loop (left) is Fowler's refactoring discipline (Refactoring 2e, 2018) and Feathers' legacy-code precondition (Working Effectively with Legacy Code, 2004), both in SOURCE-PIN §7.
