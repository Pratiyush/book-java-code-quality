## Accessibility

- Alt-text: A CI pipeline as three feedback-latency lanes (blocking PR, main, nightly) above a block-versus-warn policy strip.
- Long-description: The figure is read top-to-bottom as three feedback-latency lanes, each with a left rail naming where the stage runs (its trigger, latency, and verdict) and a track of ordered stage cards. A cost axis above the top lane reads "cheapest, most likely to fail first" on the left to "slower, more expensive" on the right. Lane 1 (On every Pull Request) is the short, blocking critical path that gives fast feedback in minutes and acts as a required status check that blocks merge; its four cards run left to right in fail-fast cheap-to-expensive order: Stage 1 Compile + format + fast lint (javac, format-check, Checkstyle 13.6.0, PMD 7.25.0; Ch 16), Stage 2 Unit tests + coverage (JUnit 6.1.0, JaCoCo 0.8.15; Ch 21, Ch 23), Stage 3 Heavier static + security (SpotBugs 4.10.2, Error Prone, Sonar; SAST, SCA, secrets; Ch 16–17, 28, 31), and Stage 4 Integration tests (Testcontainers 2.0.5; Ch 22). Lane 2 (On merge to main) sits off the PR critical path and blocks the release line, holding Stage 5 Mutation / performance (PITest 1.25.3, JMH 1.37; Ch 23) and Stage 6 Package + SBOM + sign (CycloneDX 1.6 SBOM, signing; Ch 28–29). Lane 3 (Nightly / scheduled) holds the slowest deferred checks — full DAST, deep mutation, and long performance suites — and warns-and-triages rather than blocking, noting that nightly full scans backstop incremental analysis that can miss cross-module effects. A three-card policy strip below the lanes states the gate's scope (clean-as-you-code: gate new and changed code, not whole-repo absolutes), its block axis (objective, low-false-positive, high-severity new findings, enforced as a required status check with a documented, tracked override path), and its warn axis (subjective or noisy findings, newly disclosed CVEs, security routed to a reviewer). A bottom note carries the honest center: the pipeline enforces policy but does not create quality or fix culture; a green pipeline means no detected policy violations on new code; design and logic flaws pass every automated gate and need human review (Ch 84); the two ways a gate dies are too slow and too strict; run the same build locally (./mvnw -B verify) with pinned versions; and beware false greens from a bad cache key.
