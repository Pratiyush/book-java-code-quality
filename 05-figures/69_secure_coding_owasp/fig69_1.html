Fig 69.1 — Three Java vulnerability classes: design-out hierarchy

Root cause → eliminate by construction (preferred) → mitigate when unavoidable → detect automatically. A vulnerability in the eliminate tier is impossible by construction; one only in the detect tier depends on tools not missing it.

Injection
SQL · Command · LDAP · EL/SpEL/OGNL
OWASP Top 10:2025 · A03 · CWE-89/78
Root cause
Untrusted input concatenated into an interpreter (SQL, shell, LDAP, expression language) that then executes it as code.
↓
Eliminate preferred
Parameterize — never concatenate.
PreparedStatement bind params (SQL);
JPA/Criteria parameter binding.
ProcessBuilder with fixed args (command).
Encode/parameterize LDAP, XPath, EL/SpEL/OGNL.
↓
Defense-in-depth not a substitute
Jakarta Bean Validation (structural input checks).
Note: validation alone does not stop injection — a "valid" string still injects; parameterization is mandatory.
↓
Detect automatically
SAST taint analysis (untrusted source → dangerous sink).
FindSecBugs SQL-injection rule.
Semgrep / CodeQL taint rules (Ch 31).
Insecure Deserialization
native Java serialization of untrusted data
OWASP Top 10:2025 · A08 · CWE-502
Root cause
Untrusted bytes reconstituted into live objects via readObject; gadget chains of existing classpath classes reach RCE.
↓
Eliminate preferred
Avoid native Java serialization for untrusted data entirely.
Use JSON or Protobuf without polymorphic typing enabled.
(Effective Java, Ch 5: "prefer alternatives to Java serialization.")
↓
Mitigate (unavoidable only) fallback
ObjectInputFilter (JEP 290, Java 9+) — allow-list permitted classes before deserialization.
Note: allow-list is a mitigation of an evolving gadget-chain threat, not the elimination a safe format provides.
↓
Detect automatically
FindSecBugs deserialization rule (readObject on untrusted input).
SAST taint: untrusted stream → ObjectInputStream (Ch 31).
Cryptographic-API Misuse
wrong mode · weak alg · bad randomness · hardcoded keys
OWASP Top 10:2025 · A02 · CWE-327/330
Root cause
Failure is API misuse, not a broken algorithm — wrong mode, weak choice, predictable IV/key, or bad PRNG.
↓
Eliminate via safe defaults preferred
SecureRandom (not Random/Math.random).
AES/GCM authenticated mode (not Cipher.getInstance("AES") ECB default).
bcrypt / scrypt / Argon2 / PBKDF2 for password hashing (not plain SHA).
Externalized keys; fresh random IVs per encryption.
↓
Mitigate (suppress FP with justification) fallback
Flagged MD5 may be a legitimate non-security checksum — record a justified suppression (Ch 19).
Date algorithm guidance against current OWASP/NIST (key sizes and recommendations evolve).
↓
Detect automatically
FindSecBugs: weak-algorithm, ECB, predictable-random rules.
Error Prone: @CheckReturnValue + crypto rules (Ch 18).
CryptoGuard / CogniCrypt deeper analysis.
Detects misuse patterns — not protocol or key-management design flaws.
Tier:
Eliminate — impossible by construction; no runtime dependency
Mitigate — control that can be bypassed; use only if elimination is infeasible
Detect — catches pattern-matchable signatures; misses design/logic flaws (Ch 31)
