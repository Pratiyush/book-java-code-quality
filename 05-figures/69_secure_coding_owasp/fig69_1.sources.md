## Accessibility
- Alt-text: Three columns for the Java vulnerability classes Injection, Insecure Deserialization, and Cryptographic-API Misuse, each showing root cause then a preferred Eliminate tier, a Mitigate tier, and an automatic Detect tier.
- Long-description: The figure is a three-column hierarchy; each column is one vulnerability class and reads top to bottom in order of preference. Column 1, Injection (SQL, Command, LDAP, EL/SpEL/OGNL; OWASP Top 10:2025 A03, CWE-89/78): root cause is untrusted input concatenated into an interpreter that executes it as code; Eliminate (preferred) is to parameterize and never concatenate — PreparedStatement bind params for SQL, JPA/Criteria binding, ProcessBuilder with fixed args for commands, and encoding/parameterizing LDAP, XPath, and EL/SpEL/OGNL; the next tier is defense-in-depth via Jakarta Bean Validation, noted as not a substitute because a structurally valid string still injects and parameterization is mandatory; Detect automatically uses SAST taint analysis (untrusted source to dangerous sink), the FindSecBugs SQL-injection rule, and Semgrep/CodeQL taint rules. Column 2, Insecure Deserialization (native Java serialization of untrusted data; A08, CWE-502): root cause is untrusted bytes reconstituted into live objects via readObject, where gadget chains of classpath classes reach remote code execution; Eliminate (preferred) is to avoid native Java serialization for untrusted data entirely and use JSON or Protobuf without polymorphic typing (per Effective Java, prefer alternatives to Java serialization); Mitigate (unavoidable only, a fallback) is ObjectInputFilter (JEP 290, Java 9+) to allow-list permitted classes, noted as a mitigation of an evolving gadget-chain threat rather than the elimination a safe format gives; Detect automatically uses the FindSecBugs deserialization rule and SAST taint from an untrusted stream to ObjectInputStream. Column 3, Cryptographic-API Misuse (wrong mode, weak algorithm, bad randomness, hardcoded keys; A02, CWE-327/330): root cause is API misuse, not a broken algorithm — wrong mode, weak choice, predictable IV/key, or bad PRNG; Eliminate via safe defaults (preferred) means SecureRandom (not Random or Math.random), AES/GCM authenticated mode (not the ECB default of Cipher.getInstance("AES")), bcrypt/scrypt/Argon2/PBKDF2 for password hashing (not plain SHA), and externalized keys with fresh random IVs; Mitigate (a fallback) is to suppress a false positive with justification (a flagged MD5 may be a legitimate non-security checksum) and to date algorithm guidance against current OWASP/NIST; Detect automatically uses FindSecBugs weak-algorithm/ECB/predictable-random rules, Error Prone crypto rules, and CryptoGuard/CogniCrypt, noted as detecting misuse patterns but not protocol or key-management design flaws. A bottom legend defines the three tiers: Eliminate (impossible by construction, no runtime dependency), Mitigate (a control that can be bypassed, used only if elimination is infeasible), and Detect (catches pattern-matchable signatures, misses design/logic flaws). The recurring message is that the Eliminate tier is always preferred and detection is the last layer, not the first.
- Grayscale/contrast/legibility note: Grayscale-safe — the three tiers within each column are distinguished by an uppercase text label (ELIMINATE / the mitigate label / DETECT AUTOMATICALLY), by border style (a heavy solid left-border for Eliminate, a dashed border for Mitigate, a distinct border for Detect), by the preferred/fallback text badges, and by top-to-bottom order with down-arrows — so the hierarchy reads with no reliance on hue. The Detect tier uses the warm "warn" border tone, but it renders as a distinguishable mid-gray border in print and its meaning is carried by its label and the legend, so the distinction survives in black-and-white. The legend swatches mirror the same border styles rather than relying on color. All code tokens (PreparedStatement, ProcessBuilder, ObjectInputFilter, SecureRandom, AES/GCM, Cipher.getInstance) are real, selectable monospace text understandable without syntax color, at adequate print size; contrast against the light tint fills is sufficient for low vision.
