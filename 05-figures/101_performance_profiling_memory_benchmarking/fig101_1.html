Fig 101.1 — The performance discipline loop

Measure, don't guess — the four-step loop, governed by a standing constraint.

Governing constraint Most code does not enter this loop. Premature optimization trades readability for imaginary gains — optimize only where a real target and a profile justify the cost. (Knuth; draft §"Performance as a measured requirement")
Step 1 Set a real target
  • p99 latency
  • Throughput (req/s)
  • Memory / RSS
  • Startup time
Tied to actual needs, not arbitrary numbers. "Make it fast" is unmeasurable.
→
Step 2 Profile — find the real hotspot
  • JFR — built-in, ~1% overhead, production-safe; CPU/alloc/locks/GC/IO
  • async-profiler — flame graphs; avoids safepoint bias; JFR-compatible
  • Read flame graph top-down: widest frame = where time actually goes
Profile under realistic load. Profiling shows where — not why or the fix.
→
Step 3 Benchmark the fix — JMH
  • @Benchmark + @Warmup + @Fork
  • @State non-final inputs (defeats constant folding)
  • Blackhole.consume / return result (defeats DCE)
  • Run offline; report error + environment (JDK, hardware)
JMH 1.37 (OpenJDK). Naive System.nanoTime() loops measure the optimizer, not the code.
→
Step 4 Gate against regression
  • Benchmark threshold in CI baseline
  • Load / macro testing for system-level truth
  • Environment control — same JDK + hardware
Micro is not macro: a faster JMH method may not move real end-to-end performance. Production profiling + load testing are the truth.
regression caught → reset target & re-profile
Instrument that lies #1 — the guessed hotspot
Developer intuition about where time goes is reliably wrong. A profiler under realistic load shows the truth; optimization before Step 2 wastes the effort. (Draft §"Profiling: find the real hotspot"; dossier key 102)
Instrument that lies #2 — the naive benchmark
JIT dead-code elimination, constant folding, and no warmup make a hand-rolled System.nanoTime() loop measure the optimizer. JMH's warmup + fork + Blackhole defeat each lie. (Draft §"Honest benchmarking"; JMH samples 08/09/10; dossier key 104)
Allocation hygiene — where profiling says it matters
Churn in hot paths (autoboxing, string concat in loops, unnecessary stream materialization) creates GC pressure. Escape analysis may already handle it — measure before reducing. Object pooling is usually counterproductive on modern JVMs. (Draft §"Memory and allocation hygiene"; dossier key 103)
