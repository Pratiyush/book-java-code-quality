## Accessibility

- Alt-text: A 2x2 grid of test fidelity versus input coverage, placing the unit test, integration test, property-based test, and their combination.

- Long-description: The figure, titled "Fig 22.1 — Two blind-spot axes: fidelity × input coverage" and subtitled that a unit test has two structural limits — the collaborator it models and the inputs it chose — which integration testing and property-based testing each close, has three parts.

  The main part is a 2×2 grid, shaded progressively darker toward its bottom-right. The horizontal fidelity axis runs from an in-memory model on the left to a real dependency on the right ("what the test exercises against"); the vertical input axis runs from author-chosen inputs at the top to framework-generated inputs at the bottom ("who picks the inputs"). Top-left, mock/fake with chosen inputs: Unit test (baseline), badged "two blind spots" — isolates one class, doubles stand in for collaborators, the author writes every input case; fast, deterministic, cheap; tools JUnit 6 @Test, Mockito 5.23.0, AssertJ 3.27.7. Top-right, real dependency with chosen inputs: Integration test (Testcontainers), badged "closes fidelity gap" — real Postgres or Kafka in a throwaway container catches dialect, driver, and behaviour bugs an H2 substitute or mock hides, inputs still author-chosen; tools PostgreSQLContainer / KafkaContainer via org.testcontainers:testcontainers:2.0.5, @Testcontainers + @Container. Bottom-left, mock/fake with generated inputs: Property-based test (jqwik), badged "closes input gap" — the framework generates inputs and asserts an invariant for all of them, collaborators can still be mocked so a fidelity gap may remain; tools net.jqwik:jqwik:1.10.1 (flagged maintenance mode), @Property + @ForAll + Arbitraries / @Provide, shrinks on failure. Bottom-right, real dependency with generated inputs: Integration + property-based, marked "target zone" and badged "both gaps closed" — a real container plus generated inputs gives the highest coverage at the highest cost, used where the real dependency's behaviour is the invariant under test; tools the jqwik-testcontainers integration module, costs the Docker runtime plus generated-input volume.

  A right-hand legend column explains the two axes and the cost. Fidelity axis: an in-memory fake/mock is fast and needs no Docker but tests a different engine (H2 differs from the Postgres dialect, null ordering, JSONB); a real container gives production fidelity but requires Docker everywhere and adds seconds per run. Input axis: chosen inputs (@Test / @ParameterizedTest) enumerate the cases the author imagined; generated inputs (@Property) let jqwik reach empty, boundary, emoji, and huge values automatically. A cost-rule callout, drawn from the prose, advises paying the container's cost only where the interaction with the real dependency is the behaviour under test, and the property's cost only for invariants, not every case.

  A bottom strip is an input-coverage ladder of three rungs, each shaded deeper and each widening the input space without replacing the prior. Rung 1, Example test — the author writes the input and expected output; covers cases the author thought of, misses any unlisted edge; @Test (JUnit 6). Rung 2, Parameterized test — one test body, the author supplies a table of inputs; kills copy-paste near-duplicate tests but is still author-enumerated; @ParameterizedTest + @CsvSource / @MethodSource (junit-jupiter-params, JUnit 6). Rung 3, Property-based test — the author describes the domain and invariant, jqwik generates hundreds of inputs and shrinks any failure to a minimal counterexample (with the note that jqwik 1.10.1 is in maintenance mode as of 2026); @Property + @ForAll + Arbitraries / @Provide (net.jqwik:jqwik:1.10.1). A side label marks the ladder direction as "wider input space."
