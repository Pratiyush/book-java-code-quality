Fig 45.1 — Two blind-spot axes: fidelity × input coverage

A unit test has two structural limits — the collaborator it models, and the inputs it chose. Integration testing and property-based testing each close one.

Unit test (baseline)
Isolates one class; doubles stand in for collaborators; author writes every input case. Fast, deterministic, cheap to run.

Integration + property-based
Real container + generated inputs; highest coverage, highest cost. Use where the real dependency's behaviour is the invariant under test.
jqwik-testcontainers integration module; costs: Docker runtime + generated-input volume
Input axis — who picks the inputs    ← author-chosen              framework-generated →
Fidelity axis
In-memory fake / mock: fast, no Docker; tests a different engine (H2 ≠ Postgres dialect, null order, JSONB)
Real container: production fidelity; requires Docker everywhere; adds seconds per run
Input axis
Chosen: @Test / @ParameterizedTest — you enumerate cases; covers what you imagined
Generated: @Property — jqwik generates; reaches empty / boundary / emoji / huge automatically
Cost rule (from prose): pay the container's cost only where the interaction with the real dependency is the behaviour under test. Pay the property's cost only for invariants, not every case.
