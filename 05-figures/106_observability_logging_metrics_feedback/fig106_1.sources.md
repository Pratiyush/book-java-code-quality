## Accessibility
- Alt-text: Three observability pillars — logs, metrics, traces correlated by trace ID — feed a five-step loop that turns a production incident into a test and a gate. (124 chars)
- Long-description: Two stacked sections. The top section, "The three pillars", is a row of three columns. Pillar 1, "Logs" (SLF4J facade then Logback/Log4j2), captures one event with full context: parameterized logging (log.info("user {} did {}", id, action)), structured JSON/key-value that is queryable, levels with intent (ERROR/WARN/INFO/DEBUG), and an MDC that carries the traceId from the OTel span; its limit is that logs alone miss aggregate trends and request flow across services. Pillar 2, "Metrics" (Micrometer facade, Observation API 1.10+), captures aggregate behavior over time: Counter, Gauge, Timer, DistributionSummary; tags of bounded cardinality (never a user ID); the four golden signals (latency, traffic, errors, saturation); and the Observation API producing a timer and a span in one call; its limit is that a high-cardinality tag means exploding time-series cost. Pillar 3, "Traces" (OpenTelemetry, the CNCF standard), captures how one request flowed across services: spans with W3C context propagation, an OTel Java agent that auto-instruments common libraries, population of the SLF4J MDC with traceId, and Micrometer Observations bridging to OTel spans; its limit is that the agent adds startup overhead, so only meaningful spans should be instrumented. A correlation bar beneath the three pillars reads: correlated by traceId — metric spike leads to slow trace leads to detailed log events. The bottom section, "Production feedback loop", is a row of five steps joined by right-pointing arrows: step 1 "Production — exception captured" (error tracker plus the introducing commit; marked with an accent top-bar as the loop's entry); step 2 "Triage — reproduce" (trace plus logs support the repro); step 3 "Gate — write failing test" (Ch 20, test-first from the incident); step 4 "Fix — fix + merge" (the test prevents recurrence); and step 5 "Harden — gate strengthened" (Ch 26, a new fitness function if needed). A closing "Never" callout lists three hard stops: never log a secret or PII (a password or token in a log line is a breach and a compliance violation, Ch 31); never put an unbounded value in a metric tag (a user ID as a tag produces one time series per user); and never let captured telemetry sit un-acted-upon (capture without the loop is theater — shift-right complements, never replaces, the shift-left gates).

Grayscale-safe — the three pillars share one tinted header style and are told apart by their "Pillar 1–3" labels, names, and left-to-right position; Pillar 1 (Logs) carries a slightly darker header tint (a lightness cue, not hue).
