# /draft <N> — Step 4 onward: draft a chapter and run the gates

You are the **drafting conductor** for chapter key `$ARGUMENTS`. You drive Step 4 (draft) and then the full gate chain through to the human approval gate (Steps 4a → 12). You do **not** approve or commit — that is `/review-chapter` re-runs and the human gate. **`PIPELINE.md` is the authoritative step list and HARD-gate enumeration; the numbers below mirror it — where they disagree, PIPELINE.md wins.**

## Hard preconditions (verify before drafting)
1. `FINAL_INDEX.md` is **human-confirmed** and this key is in it. If the index is still PROPOSED, **stop** — drafting is hard-blocked.
2. The dossier `02-research/NN_slug/NN_slug_RESEARCH.md` is banked **and** its step-2 source-trace audit passed **@ `the pins in SOURCE-PIN.md`** (`NN_slug_VERIFY.md`). For any key researched against an earlier or unpinned version of `the pinned authority set (00-strategy/SOURCE-PIN.md)`, this means the re-verification has run — if not, stop and route to `/research`.
3. Step 0 holds: `the per-tool fetch dirs in SOURCE-PIN.md` is on-pin (the local copy may be **ephemeral**; if absent or off-pin, run `scripts/ensure_source_pin.sh` — or `/pin-source` — before any verification read).
4. **Pre-draft plan is fixed** (PIPELINE.md Step 4 fixes these *before* a word is drafted): (a) the 2–3 sentence **chapter promise**; (b) the **worked-example spec** per `one runnable, enterprise-grade module/example per chapter; the displayed snippet is a tag-region inside the compiled file; built green` and `templates/EXAMPLES-GUIDE.md` *(technical profile — see BOOK-TYPE-PROFILES.md; this is the companion `08-companion-code/NN_slug/` runnable module realizing this chapter's demo from `DEMO-CATALOG.md`; book types with the build/compile gate turned off substitute a worked scenario / thought experiment)*; (c) the **figure plan** — which designed diagrams + native screenshots the chapter carries, per the per-chapter image budget in `per-chapter image budget; designed diagrams authored as HTML then rendered to PNG (never image-generated); tool/IDE screenshots captured` (`GUIDELINES §8`). If any of these is unfixed, fix it before drafting.

## Step 4 — draft (drafter agent)
Run the `drafter` agent to write `03-drafts/NN_slug/NN_slug_v1.md` from `templates/CHAPTER-TEMPLATE.md`, using the `book-law` and source-verification skills.
- **Draft from the dossier ONLY.** Every rule IDs, config/ruleset keys, tool flags, API signatures, GAV coordinates, version numbers, benchmark figures, and quoted claims and every snippet must already be in the verified dossier with a recorded source path/citation. The drafter introduces **no** new untraceable facts; if the chapter needs a fact the dossier lacks, stop and send it back to `/research` — do not invent it.
- Snippets ≤9 lines, each carrying its verified source path.
- Hold the voice of `VOICE-GUIDE.md` (the locked voice in 00-strategy/VOICE-GUIDE-JAVA-QUALITY.md). Honor neutrality (neutral comparative survey (see 00-strategy/NEUTRALITY.md): each option gets its strongest case and its hardest limitation; no crowning; banned: better than / unlike X / superior / beats / the problem with X) and give every feature a when-NOT-to-use.
- **No word-count target.** Length follows substance.

## The gate chain (run in order; HARD gates block — PIPELINE.md numbering)
- **Step 4a — PRE-CHECK (HARD):** the fast, mostly-greppable pre-gate that bounces an obviously-broken draft before any expensive agent reads it — banned-phrase grep (the neutrality + voice + self-narration lists) + `lint_citations.sh` (two-tier plain-text URLs, ≥1 source outside `the pinned authority set (00-strategy/SOURCE-PIN.md)`) + `check_snippets.sh` (every displayed snippet resolves to a real ≤9-line tagged region). Also greps em-dash density and comparative section titles.
- **Step 4b — EXAMPLE-BUILD (HARD) (technical profile — see BOOK-TYPE-PROFILES.md; book types with the build/compile gate turned off drop this step):** hand off to `/example <N>` to build the companion module, run the CODE-REVIEW sub-gate, and record `NN_slug_EXAMPLE.md` + `NN_slug_CODEREVIEW.md` BEFORE VERIFY. The module (`08-companion-code/NN_slug/`) must (1) compile and test green via `./mvnw -B verify` (lint-clean, at `the pins in SOURCE-PIN.md`, `Java 21+ (21 LTS anchor, 25 LTS forward)`) AND (2) pass CODE-REVIEW (the `code-reviewer` agent). Run this EARLY — it is the cheapest objective signal; a red build OR a CODE-REVIEW FAIL bounces the chapter before clarity/audit/score are spent on it.
- **Step 4c — CAPTURE (soft, feeds Step 9) (technical profile — see BOOK-TYPE-PROFILES.md; book types with the build/compile gate turned off drop this step):** with the module green, run it and capture the chapter's **native UI** screenshots under `05-figures/NN_slug/` with a sidecar manifest. Third-party-tool UIs are never captured — a designed diagram (Step 9a) replaces them. A chapter that plans no UI screenshot skips this.
- **Step 5 — VERIFY (HARD):** `verify_sources.sh` + `lint_citations.sh` + `source-verifier` over the draft, re-traced against the source *(technical profile: re-traced against the compiled code)*. Every fact re-traced to the pin; two-tier plain-text citations present. → `NN_slug_VERIFY.md`.
- **Step 6 — Technical clarity review (HARD):** `tech-clarity-reviewer` agent — is the mechanism explained cleanly?
- **Step 7 — AUDIT (HARD):** `auditor` agent — one cold read for NEUTRALITY + voice + authenticity, plus a scripted banned-phrase pre-pass (the `neutral comparative survey (see 00-strategy/NEUTRALITY.md): each option gets its strongest case and its hardest limitation; no crowning; banned: better than / unlike X / superior / beats / the problem with X` banned list). A sharp reader must not be able to tell a machine wrote it. Fail → revise and re-audit.
- **Step 8 — Scorecard (HARD):** `chapter-scorer` against `SCORING.md` → `NN_slug_SCORE.md`. Ship bar: **all floors PASS** (A NEUTRALITY / B HONEST-LIMITATIONS / C SOURCE-TRACE + COMPILE + CODE-REVIEW), **AND** the five clusters (the five quality clusters in 00-strategy/SCORING.md) meet `the numeric ship bar in 00-strategy/SCORING.md`. Miss on cluster quality → **bounded lift loop** (≤3 in-bounds passes on the weakest cluster, re-score each pass; still short → cut candidate, flag to `09-flags/`).
- **Step 9 — FIGURES (HARD for any planned figure):** produce the diagrams fixed in the figure plan. Hand off to `/figure <N>` — **9a FIGURE-DESIGN** authors each load-bearing conceptual diagram as HTML (`05-figures/NN_slug/figNN_x.html` + a source-trace sidecar; every label/number traces to the pin) and **9b FIGURE-RENDER** renders HTML→cropped PNG via `05-figures/_assets/render.mjs` and runs the figure-accuracy + neutrality + IP check. Designed diagrams are **never** generated by an image model. The per-chapter image budget is governed by `per-chapter image budget; designed diagrams authored as HTML then rendered to PNG (never image-generated); tool/IDE screenshots captured` (`GUIDELINES §8`); a no-figure chapter skips 9a/9b.
- **Step 10 — Reconcile (HARD):** `reconcile_facts.sh` + `reconciler` agent vs the ledger — canonical names, rule IDs, config/ruleset keys, tool flags, API signatures, GAV coordinates, version numbers, benchmark figures, and quoted claims, and version pins agree across chapters; hand-off chain intact.
- **Step 11 — Maintainer:** `book-maintainer` updates `LEDGER.md` §1 and `CHAPTER-TRACKER.md`; captures learnings.
- **Step 12 — STOP at the human approval gate (HARD).** The human reads the chapter **together with its rendered figures and the gate reports** and decides. Do not approve, finalize, or commit — approve → `04-approved/` and the one-commit-per-chapter (chapter + `_SCORE.md` + figure assets) is **Step 13**, gated by this human stop. Leave the chapter at the gate for the human.

## Tooling honesty
The scripts (`verify_sources.sh`, `lint_citations.sh`, `check_snippets.sh`, `reconcile_facts.sh`) and agents may be newly authored. Where one is not yet built, run that gate as a documented manual procedure and say so — *(technical profile — see BOOK-TYPE-PROFILES.md)* Step 4b is unblocked only once the build wrapper is committed at `08-companion-code/` root and a `Java 21+ (21 LTS anchor, 25 LTS forward)` baseline is installed so `./mvnw -B verify` can run. The figure renderer (`05-figures/_assets/render.mjs`, Step 9b) is typically the one exercised piece of tooling; most other tooling is manual until confirmed. The local copy `the per-tool fetch dirs in SOURCE-PIN.md` may be **ephemeral** — re-run `scripts/ensure_source_pin.sh` (or `/pin-source`) on a fresh machine before any verification read.

## Report
Per gate: PASS/FAIL, lift-loop passes if any, final scorecard, open flags. End at the human gate. Close with **"Learnings & pipeline suggestions"**.
