State verification

The test checks the outcome — the state of the system after the method runs. Used with stubs and fakes; doubles supply inputs, the assertion checks outputs.

Applies to: Dummy · Fake · Stub (and Spy used as stub)
Strength: Tests survive refactoring of internal call structure — the assertion is on observable behaviour, not how it was achieved.

Behaviour verification

The test asserts an interaction — that the unit made the right calls on its collaborator. "Mock objects always use behaviour verification, while a stub can go either way."

Applies to: Mock (and Spy when verifying calls)
Cost: Couples the test to the implementation's call structure — breaks on a refactor that changes how the code collaborates, even when the observable behaviour is unchanged.

Quality guard
STRICT_STUBS (default):
unused stub → UnnecessaryStubbing
arg mismatch → PotentialStubbingProblem
Built-in quality guard: MockitoExtension defaults to Strictness.STRICT_STUBS — a dead stub fails the build (UnnecessaryStubbingException); an argument mismatch on a stubbed call raises PotentialStubbingProblem. Do not mock value objects (records / immutable data) — use the real instance.
