Fig 75.1 — The quality pipeline: a fail-fast portfolio of fitness functions, split by feedback latency

Each stage is one fitness function; the pipeline is the portfolio. Order cheap → expensive so the most-likely-to-fail checks fail first. Keep the blocking PR path short; defer the slow, expensive checks to main and nightly so they never fall on a developer waiting to merge.

Scope — clean as you code

Gate new and changed code, not whole-repo absolutes. A "zero findings / 80% coverage" rule on a legacy codebase blocks every PR on debt it did not create; new-code scope makes the gate adoptable and assigns ownership to what a developer touches. (Ch 19, Ch 34)

Block

Objective, low-false-positive, high-severity new findings — compile, failing tests, new-code coverage, new high-severity static/security, new secrets. Enforced as a required status check in branch protection, with a documented, tracked override path. (Ch 35)

Warn → triage

Subjective or noisy findings (so red always means something real); newly disclosed CVEs in unchanged code; security routed to a reviewer for an exploitability judgment. (Ch 19, Ch 32)

The pipeline enforces policy; it does not create quality. A green pipeline means "no detected policy violations on new code" — design, logic, and authorization flaws pass every automated gate and still need human review (Ch 84). The two ways every gate dies: too slow, so it is bypassed, and too strict, so it is disabled. A gate the team routes around is a net negative. Run the same build developers run locally (./mvnw -B verify) with pinned tool versions so it cannot drift, and beware false greens from a bad cache key (clean build on main backstops).
