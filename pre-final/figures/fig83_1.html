Fig 36.1 — The release-quality loop: shift-left gates ↔ shift-right release quality

Every gate lowers the odds a defect ships; none reaches zero. Release quality assumes one slips, limits its blast radius, and feeds the lesson back into the gates — one continuous cycle, not two phases.

→ SHIFT-LEFT — catch defects before merge  the gates of Ch 33–35
→ SHIFT-RIGHT — limit damage & learn in production  release quality (this chapter)
1 Release gates

Final, artifact-level checks — not re-litigating code quality.

  • All CI gates green on the release commit (Ch 33–35)
  • SBOM generated; artifact signed / attested (SLSA / cosign, Part VII)
  • semver bump + changelog (consumers know what changed)
  • Smoke tests vs a staged build
→green &
signed
2 Progressive delivery

Limit the blast radius of the change that slips.

  • Canary — small % of traffic; watch error / latency; promote or roll back
  • Blue-green — two envs; rollback is a traffic switch
  • Feature flags — deploy dark, release gradually, kill-switch (Ch 35)
a defect slips → hits 1%, not 100%
→gradual
rollout
3 Production & monitor

Watch the running release — the signals progressive delivery promotes on.

  • Error tracking (Sentry-class) surfaces a regression
  • Metrics / alerts + SLOs (Part XIII)
  • Continuous CVE re-scan of shipped deps (Dependency-Track, Part VII)
→regression
found
4 Promote or roll back

Decide on the signals — recovery in seconds, not a panicked redeploy.

  • Healthy → promote to 100%
  • Bad → flag flip / traffic switch / roll back
  • Stateful change → backward-compatible (expand–contract) migration, not raw rollback
Decouple deploy from release. Deploying code (the binary in production) and releasing a feature (turning it on for users) are separate acts. Code deploys continuously, dark behind flags; features release gradually via canary or flag rollout — which is what makes shipping safe and frequent at once. The same practices that make releases frequent (canary, flags, fast rollback) are the ones that make them safe; frequent and safe are produced by the same mechanism, not traded off (DORA, key 02).

The new gate is itself shift-left for the next change — ↩  feeds back into the gates (Stage 1). What production catches becomes a gate, so the pipeline gets better at the failures that actually happen. The quality cycle closes.
Measured by DORA stability metrics — change-failure rate and failed-deployment recovery time are release quality; canary, flags, and fast rollback drive them down (a bad change caught at 1% with a flag flip is near-zero impact, seconds of recovery) (Part X, key 85).
The honest ceiling — this loop limits damage; it does not prevent defects
Safe release ≠ good code. Damage control, not prevention — prevention is the rest of the book (types, tests, analysis, review).
Needs infra + good metrics. Canary analysis is blind without solid observability (Part XIII); a small internal app may not need it.
Flags become debt. Stale flags → complexity + test-matrix explosion; they need a removal discipline after rollout.
Rollback isn’t always clean. DB / stateful changes can’t reverse — design backward-compatible migrations.
Feedback only helps if acted on. Error-tracking noise nobody triages is theatre (the vanity-metric trap).
Gates verify the artifact, not the design. The logic flaw is caught upstream by review (Ch 37), not by smoke tests.
