Fig 55.1 — Architecture-enforcement spectrum

Convention → ArchUnit rule → JPMS compiler boundary: enforcement strength rises with adoption cost. Choice is a trade-off, not an upgrade path.

LEVEL 0 Naming convention — agreed in the wiki enforced by: discipline (nobody)
strength: none cost: free
what holds it Team memory and good intentions; shared agreement recorded in a wiki or ADR.
honest limit Invisible to the compiler and the build — a single import under deadline pressure violates it silently; the architecture erodes one import at a time.
↓
make the decision executable → test phase
LEVEL 1 ArchUnit rule — architecture as a JUnit test enforced by: JUnit / CI (test phase)
strength: fails the build on violation cost: one test dependency; post-compile
what it catches Layer-direction violations (domain → web import); cycles (slices().beFreeOfCycles()); naming, annotation constraints, coding rules (NO_CLASSES_SHOULD_ACCESS_STANDARD_STREAMS).
honest limit Bytecode-only: dependencies via reflection, Class.forName, DI wiring, or ServiceLoader are not edges ArchUnit sees — a routed violation passes. Only catches what is encoded; runs post-compile (slower than the compiler).
@AnalyzeClasses(packages = "org.acme.storefront", importOptions = DoNotIncludeTests.class)
noClasses().that().resideInAPackage("..domain..")
  .should().dependOnClassesThat().resideInAPackage("..web..")
// FreezingArchRule.freeze(rule) → ratchet for legacy codebases
↓
move boundary into the compiler → compile + runtime
LEVEL 2 JPMS module-info.java — compiler-enforced boundary enforced by: javac + runtime (Java 9+, JEP 261)
strength: javac refuses to compile the violation cost: real modularization migration
what it catches Any access to a package not declared exports in the module-info.java fails javac and the runtime — structural, not after-the-fact. Strong encapsulation at the module level.
honest limit Modularizing a legacy application with non-modular dependencies is a real migration; many teams rationally stay on the classpath. Coarser-grained than ArchUnit (module boundary, not package/annotation); ArchUnit for fine-grained rules, JPMS for coarse boundaries that warrant the cost.
module org.acme.storefront.domain {
  exports org.acme.storefront.domain.model; // explicit; web pkg not exported
  requires org.acme.storefront.persistence;
}

When to reach for each
► Stop a documented architecture from eroding: Level 1 — ArchUnit rule in CI.
► Cardinal-sin check (no cycles): slices().beFreeOfCycles() — cheapest, enable first.
► Legacy codebase adoption: FreezingArchRule.freeze(rule) then drive baseline down.
► Coarse structural encapsulation that warrants migration cost: Level 2 — JPMS.
► What cannot be expressed as a rule: code review & architectural judgment.
Alternatives (neutral)
jQAssistant — Cypher queries over a Neo4j graph of the codebase; suited to ad-hoc exploration.
Spring Modulith — module verification for Spring apps.
Deptective — compiler plugin.
JDepend — afferent/efferent coupling metrics; measurement, not a gate.
Each serves its niche; none crowned.
The fitness-function frame — these mechanisms, and every gate in this book, are the same thing
Ford / Parsons / Kua / Sadalage, Building Evolutionary Architectures: “any mechanism that provides an objective integrity assessment of some architectural characteristic(s)” — run on every build or deploy to protect properties as the system evolves.

One designed portfolio of atomic/holistic, static/dynamic, triggered/continuous guards for the characteristics a system must protect. Limits: only protects measurable characteristics; too many strict functions ossify (breed ignore-annotations); functions are code that rots — design deliberately and maintain.
