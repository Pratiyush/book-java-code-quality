Fig 106.1 — The three pillars of observability and the production feedback loop

Logs · Metrics · Traces — correlated by trace ID — feed the loop that turns incidents into tests.

The three pillars (§A–§B)
Pillar 1 Logs SLF4J facade → Logback / Log4j2

One event, full context

  • Parameterized: log.info("user {} did {}", id, action)
  • Structured JSON / key-value — queryable
  • Levels with intent: ERROR / WARN / INFO / DEBUG
  • MDC carries traceId from OTel span

Alone: misses aggregate trends, request flow across services

Pillar 2 Metrics Micrometer facade (Observation API 1.10+)

Aggregate behavior over time

  • Counter · Gauge · Timer · DistributionSummary
  • Tags: bounded cardinality — never a user ID
  • Four golden signals: latency / traffic / errors / saturation
  • Observation API → timer and span in one call

High-cardinality tag = exploding time-series cost

Pillar 3 Traces OpenTelemetry (OTel) — CNCF standard

How one request flowed across services

  • Spans + W3C context propagation
  • OTel Java agent auto-instruments common libs
  • Populates SLF4J MDC with traceId
  • Micrometer Observations bridge to OTel spans

Agent adds startup overhead; instrument meaningful spans only

Correlated by traceId — metric spike → slow trace → detailed log events

Never Log a secret or PII — a password or token in a log line is a breach and a compliance violation (Ch 31). Put an unbounded value in a metric tag — a user ID as a tag produces one time series per user. Let captured telemetry sit un-acted-upon — capture without the loop is theater; shift-right complements, never replaces, the shift-left gates.
