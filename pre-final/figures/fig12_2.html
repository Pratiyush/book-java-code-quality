Fig 10.2 — try-with-resources: suppressed vs masked exceptions

JLS SE 21 §14.20.3 · Effective Java 3e Item 9 · Throwable.addSuppressed/getSuppressed (JDK 21 API)

Old pattern: try/finally BUG: masks original exception
1
Body executes — throws E1 (the real failure)
e.g. read error on the stream
2
finally { resource.close(); } runs — close() throws E2
3
E2 propagates; E1 is silently discarded.
The real failure disappears. Stack trace is gone.
✕
With multiple resources and nested finally: each additional close can mask the previous exception. "The code is even more inscrutable." — EJ Item 9
E1 (root cause) lost forever No way to recover or even log the original failure.
try-with-resources CORRECT: suppresses, never masks
1
Body executes — throws E1 (the real failure)
2
Resources close in reverse order (LIFO) — JLS §14.20.3.
Last opened closes first; each close runs on every path.
3
If close() throws E2: E1.addSuppressed(E2) is called automatically.
E1 propagates; E2 is attached, not lost.
✓
Retrieve via e.getSuppressed() — the close exception is still accessible for diagnostics.
E1 propagates · E2 attached as suppressed Root cause preserved. Caller can inspect both. JDK 21 API: Throwable.getSuppressed()
Close order
LIFO (reverse of declaration). In try (A a = …; B b = …), b.close() runs before a.close(). JLS §14.20.3.
Every path
close() runs on normal completion, on a body throw, and even if a later resource's initialization fails. JLS §14.20.3.
AutoCloseable contract
AutoCloseable.close() — not required idempotent. Closeable (its subtype) — required idempotent. Write custom implementations idempotent anyway. JDK 21 API.
Linter enforcement — resource leaks trigger: Sonar java:S2095 · PMD CloseResource · SpotBugs OS_OPEN_STREAM / OBL_UNSATISFIED_OBLIGATION · Error Prone MustBeClosed / StreamResourceLeak. These fire when a resource escapes the try-with-resources header (created in the body, returned from a factory, or stored in a field).
