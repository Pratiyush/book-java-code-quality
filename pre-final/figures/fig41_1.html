Fig 20.1 — The test pyramid

Two rules (Vocke / Cohn): “Write tests with different granularity” and “The more high-level you get the fewer tests you should have.”

The pyramid: shape encodes cost / speed / confidence
UI / End-to-End
Full-stack behaviour; slowest, costliest to run and maintain
fewest
Integration / Service
Application “correctly interacts with external dependencies like databases, filesystems, and separate services”
fewer
Unit
“Focused, isolated tests of individual code components with the narrowest scope.” Fast, cheap, plentiful.
most
Build expression — Maven Surefire (test phase) runs unit tests; Maven Failsafe (verify phase) runs *IT integration tests. That split is the pyramid in the build.
Anti-pattern: the ice-cream cone
Unit — few
Integration
E2E — many (widest base)
“Excessive high-level tests that become maintenance nightmares and run slowly” (Vocke verbatim). E2E tests are “notoriously flaky and often fail for unexpected and unforeseeable reasons.”
Vocke rule 1 (verbatim)
“Write tests with different granularity.”
Vocke rule 2 (verbatim)
“The more high-level you get the fewer tests you should have.”
Heuristic, not a law
Vocke’s layer names are “deliberately fuzzy” and the shape is a rule of thumb, not a 70/20/10 quota. Alternative weightings (testing trophy, honeycomb) put more weight on integration tests. Each is a different trade-off; none is crowned.
Solitary vs. sociable
Solitary: stubs all collaborators for isolation.
Sociable: lets the unit interact with real collaborators where practical. Root of the mocking debate → Ch 21.
Vocke prescription (verbatim)
“Test observable behaviour, not implementation details.”

Sources: Cohn, Succeeding with Agile (2009) — origin; three layers Unit/Service/UI.
