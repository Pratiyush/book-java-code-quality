Fig 71.1 — Secrets detection: a ladder of catches, cheapest first — and why finding is already too late

Each later stage backstops the one before, because the earlier ones are bypassable. Only the first stage prevents the leak; past the remote, the secret is compromised — the fix is to rotate, not delete.

STAGE 1 · LOCAL, BEFORE COMMIT
Pre-commit hook
prevents the leak Scans the staged diff; blocks the commit before the secret ever lands. The cheapest catch — and the only stage that prevents the leak entirely.
Bypassable: git commit --no-verify skips it → needs a backstop.
→
STAGE 2 · AT THE PLATFORM, ON PUSH
Push protection
blocks at the remote Platform secret scanning rejects the push if it carries a known secret pattern — the last line before the credential is on the server.
STAGE 3 · IN CI, AFTER PUSH
CI gate
catches what slipped Runs the scanner in the pipeline to catch a secret from a developer who skipped the hook. Already pushed — this fails the build, it does not un-leak.
→
STAGE 4 · PERIODIC, OVER HISTORY
History scan
finds the already-committed Sweeps past commits for credentials committed long ago. Detection only — whatever it finds was reachable in the history the whole time.
← can still prevent the leak (local)
already out — detection, not prevention →
If it reached a remoteA committed secret is a compromised secret — detection is not remediation
Assume
Once on a remote (especially public), assume it was scraped — bots pull new commits in seconds.
→
Necessary, not sufficient
Delete from HEAD · rewrite history (git filter-repo, BFG) — does not un-leak what was already pulled.
→
The only real fix
ROTATE the credential — revoke the exposed one, issue a new one. Non-negotiable; history rewrite is secondary.
Detection is the net; prevention is the floor. The real fix is to never put secrets in source at all — externalized config and a secret manager (environment variables, Vault, cloud secret stores). Where one escape is catastrophic, invest in the floor, not only the net. Secrets detection is the safety net under that discipline, not a substitute for it.
How it detects: regex/rule patterns (AWS keys, tokens, private keys) + entropy analysis (random-looking strings); some tools verify against the provider.
Tools (crown none): gitleaks · TruffleHog · platform secret scanning.
False positives: entropy flags UUIDs, hashes, test fixtures → allow-list (Ch 19, key 39).
Backstop chain: pre-commit & CI parity (key 82); secret managers → ops chapters.
